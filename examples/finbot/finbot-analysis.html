Agent Remit (as declared)
FinBot is declared as an autonomous invoice processing assistant for CineFlow Productions: it receives vendor invoices, validates them against registered vendor records, makes approve / reject / escalate decisions, and records outcomes back to the invoice database. The remit authorizes a fixed five-tool set — get_invoice_details, approve_invoice, reject_invoice, request_human_review, and detect_fraud_patterns — and confines counterparties to registered vendors (as data, not as instructors), CineFlow finance admins (config only), and the OpenAI API (inference only). The remit MUSTs are specific: fraud detection always on, vendor must be pre-registered and approved, invoices above manual_review_threshold require human sign-off, and the decision logic is explicitly declared to be code-defined and not runtime-redefinable by any user, vendor, or API caller.
Agent Structure (as observed)
Python 3.11 Flask application (Flask==3.1.1, flask-sqlalchemy, flask-cors) deployed under Gunicorn on Render, with SQLAlchemy-backed SQLite (or PostgreSQL via DATABASE_URL). A single FinBotAgent class in src/services/finbot_agent.py orchestrates an OpenAI gpt-4.1-mini function-calling loop (max 5 iterations) over the five remit tools, plus a rule-based _fallback_processing path that fires when the OpenAI client init fails. The system prompt is assembled at every call from the live finbot_config table — including a free-form custom_goals column that is concatenated verbatim under a CUSTOM GOALS (OVERRIDE ABOVE IF CONFLICTING) header. The HTTP surface is three Flask blueprints under /api — user_bp, vendor_bp, admin_bp — and the entire admin_bp (goals update, config update, vendor-trust update, reprocess, CTF flag readout) is exposed without any authentication, authorization, or rate-limiting middleware; CORS is open to *.
Behavior Summary

The dominant pattern is policy declared in prose, zero code-level enforcement, joined to an unauthenticated admin surface that lets anonymous callers rewrite the agent's goals at runtime. Every MUST / MUST NOT clause in the remit that matters — fraud detection always on, manual-review threshold respected, decision logic not redefinable at runtime, no instructions from invoice content — is enforced only inside the LLM system-prompt string, with no Python-side gate on _approve_invoice, no signature or auth check on POST /api/admin/finbot/goals, and no policy check on the fraud_detection_enabled toggle.

That string is then assembled at every process_invoice call from the live finbot_config row, including the writable custom_goals column — producing a one-hop chain from the internet to persistent agent compromise: any anonymous POST to /api/admin/finbot/goals writes natural-language overrides into every subsequent decision, and the same agent also receives unsanitized vendor invoice descriptions into its context window and is authorized to set payment_processed=True. The CTF walkthrough at docs/FinBot-CTF-walkthrough-goal-manipulation.md documents the chain as designed; the scan's job here is to record that the remit and the implementation do not agree on whether that design is the intended posture.

Maturity Scoring Rubric

Every score above is based on this scale. A score is a snapshot of observable posture — not a verdict on the people or team behind the system.

Score Label Meaning
5 Exemplary Best-in-class; automated, continuously tested, reference quality. Rarely achieved in shipping systems.
4 Strong Comprehensive controls, active management, minor gaps. Production-ready.
3 Established Documented controls consistently applied; known gaps accepted. A respectable baseline.
2 Partial Some controls exist but coverage is incomplete; key gaps remain.
1 Ad hoc Informal or inconsistent measures; relies on individual judgment.
0 Absent No evidence this category is addressed at all.
Weighting: Zero Trust counts at 25 percent (double); the other five categories each count at 15 percent. The weighted overall sums score × weight across all six categories to produce a 0.0–5.0 scalar.
