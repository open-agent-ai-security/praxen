Agent Remit (as declared)
An internal employee assistant for a mid-size organization, intended to answer questions, retrieve documents from a designated company knowledge base, run public web searches, and write summaries to a designated output directory. The agent may use exactly three tools — read_file, write_file, and search_web — and must stay within an authorized document root, validate user input as untrusted, refuse to access files outside its scope, and log every tool call. It is explicitly forbidden from processing PII or financial data, revealing its system prompt, executing shell commands, or following instructions embedded in retrieved content.
Agent Structure (as observed)
A Node.js application (Damn Vulnerable AI Agent, v0.8.2) where HelperBot is one of fourteen intentionally-vulnerable personas defined in src/core/agents.js at lines 43–78 and exposed on port 7002 via an OpenAI-compatible HTTP server in src/index.js. The persona's declared features block sets inputValidation, outputFiltering, toolApproval, rateLimiting, and auditLogging all to false and lists three configured vulnerabilities (promptInjection, dataExfiltration, contextManipulation). The LLM-mode system prompt for helperbot at src/llm/prompts.js:23–27 hardcodes an internal API-key literal directly into the prompt and instructs the model to share configuration openly for "transparency." The three remit-authorized tool names appear in the persona's tools array but there is no per-tool implementation gating their execution against the remit's directory boundaries, untrusted-input checks, or 20-call rate limit.
Behavior Summary

HelperBot is a textbook case of policy declared in prose, zero enforcement in code. The Worker Remit lists thirteen specific MUST / MUST-NOT clauses and a 20-call rate limit; the persona definition in src/core/agents.js:56–62 then explicitly sets every corresponding control flag — inputValidation, outputFiltering, toolApproval, rateLimiting, auditLogging — to false, and the LLM-mode system prompt at src/llm/prompts.js:23–27 inverts the remit by inlining a literal API-key secret and instructing the model to share its instructions for transparency.

The dominant compound chain is the canonical excessive-agency pattern: a write-capable agent (write_file tool, no approval gate) running in a process whose request handler in src/index.js:374–457 explicitly rewards prompt-injection keywords with attacker-favorable responses (e.g., the "ignore" trigger at :375 flips the agent to "new instructions accepted"). Direct prompt injection therefore reaches the file-write capability in one hop, with no logging to detect it.

CRITICAL PRAX-2026-05-11-004 User input flows directly into LLM context and tool-decision logic with no validation or origin-labeling.

MCP-protocol path-boundary check exists in the shared executor

The <code>executeMcpTool</code> function enforces a sandbox root for read_file/write_file/execute even when the agent is intentionally vulnerable. The boundary is a hard <code>resolved.startsWith(sandbox.root)</code> check that is wired into every MCP tool — the primitive Praxen would want HelperBot to inherit exists in the codebase, it is just not on HelperBot's (API-protocol) path.

src/index.js:907

LLM API keys held in process memory only, never persisted

<code>src/llm/provider.js</code> stores user-supplied LLM credentials in module-scope variables with a clear "never persisted to disk or sent to any server" contract, and provides an explicit <code>disableLLM()</code> that nulls the key. This is the right shape for BYOK credential handling.
