BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy-declared-in-prompt, zero code-level
enforcement: every prohibition the remit treats as a hard MUST NOT — no DML,
no DDL, no admin commands, no multi-statement — exists only as the prose line
`DO NOT make any DML statements` in `agent_toolkits/sql/prompt.py`, while the
execution path `QuerySQLDatabaseTool._run` -> `run_no_throw` -> `_execute`
wraps any LLM-generated string in `text()` and runs it inside a committing
`engine.begin()` transaction with no statement-type gate. The remit's own
clause that "the agent's own logic MUST NOT be the sole enforcement point" is
therefore violated by construction — the LLM prompt is the only barrier in the
code Praxen can see. This is a documented, maintainer-acknowledged design: the
`create_sql_agent` docstring warns the agent runs arbitrary SQL and pushes the
real controls (read-only roles, server-side timeouts) to the operator, so the
divergence is between the library's in-code posture and the safe deployment
the remit assumes. Genuine in-code mitigations — the `max_iterations=15` loop
cap and `max_string_length` result truncation — are real and earn credit, but
they bound runaway loops and output size, not the write prohibition.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-06-15-001  The no-DML / no-DDL / no-admin prohibition exists only
            as an LLM prompt line; the query tool executes any SQL the model
            emits with no code-level gate.
      Action 1: Do not rely on the prompt: enforce the write prohibition
                outside the agent by connecting `SQLDatabase` with a
                least-privilege read-only DB role, as the `create_sql_agent`
                docstring already advises.
      Action 2: If in-process defense is wanted, add a statement-type gate
                (parse/inspect the command in `_execute` or a
                `QuerySQLDatabaseTool` wrapper) that rejects anything but read
                statements before execution.

  PRAX-2026-06-15-002  Query results and schema rows flow into LLM context
            unlabeled, giving injected row/table-name content a one-hop path
            to drive arbitrary SQL through the ungated query tool.
      Action: Treat all DB-returned content (rows, table/column names, sample
              rows) as untrusted in prompt construction and pair it with the
              read-only role from PRAX-2026-06-15-001 so an injected write
              cannot execute.

  PRAX-2026-06-15-003  The remit's "agent logic MUST NOT be the sole
            enforcement point" for the write prohibition is violated by
            construction — the LLM prompt is the only barrier in the scoped
            code.
      Action: Make a non-LLM layer authoritative for the write prohibition — a
              read-only DB role is the lowest-effort durable control; document
              it as required (not optional) for any deployment that exposes
              untrusted questions.
