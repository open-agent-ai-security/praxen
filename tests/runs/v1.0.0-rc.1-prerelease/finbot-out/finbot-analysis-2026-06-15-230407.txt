BEHAVIOR SUMMARY
------------------------------------------------------------------------------
FinBot is a deliberately-vulnerable demo whose dominant pattern is policy
declared in a natural-language prompt with zero code-level enforcement,
compounded by an unauthenticated mutation surface for that policy. Every remit
guarantee — fraud detection must run, the manual-review threshold is a hard
ceiling, vendor content must not alter decisions, only registered vendors get
paid — lives only as English in the system prompt or as bypassable
business-logic heuristics; none is a deterministic gate. The catastrophic
chain is end-to-end: an unauthenticated attacker POSTs to
/api/admin/finbot/goals to inject "OVERRIDE" goals or to
/api/admin/finbot/config to disable fraud detection, then submits an invoice
whose description — vendor-controlled text that flows straight into the LLM
context — carries business-manipulation language that the
_fallback_processing() "business context" scorer treats as grounds to
auto-approve above the manual-review threshold, completing goal-hijack to
autonomous payment with no human checkpoint and no audit log.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-06-15-001  Unauthenticated goal-hijack to autonomous-payment chain
            — any caller can rewrite FinBot's goals, disable fraud detection,
            and trigger payment of a manipulated invoice with no human
            checkpoint.
      Action 1: Put all /admin/* routes behind authenticated, role-scoped
                middleware and remove the natural-language goal-mutation
                endpoint entirely; goal changes must require a code
                deployment, not a runtime POST.
      Action 2: Make payment a deterministic gate: _approve_invoice must
                verify vendor approved-status, enforce the manual-review
                ceiling, and require a passing fraud check before setting
                payment_processed.

  PRAX-2026-06-15-002  Fraud detection is never enforced as a gate before
            approval — the approve path sets payment with no required
            preceding fraud check.
      Action: Make a passing fraud-detection result a hard precondition inside
              _approve_invoice rather than an optional LLM tool call.

  PRAX-2026-06-15-003  Fraud detection is a runtime-flippable config boolean —
            any unauthenticated caller can disable it, which the remit forbids
            absolutely.
      Action: Remove fraud_detection_enabled as a runtime toggle; fraud
              detection must always run and not be reachable by configuration.

  PRAX-2026-06-15-004  Vendor-controlled invoice description flows unsanitized
            into the LLM context and decision logic — an indirect
            prompt-injection sink the remit explicitly forbids.
      Action 1: Treat invoice description as untrusted data: never feed it as
                instructions and remove it from the auto-approve scoring path.
      Action 2: Replace the description-phrase business-context heuristic with
                verified data-source checks rather than vendor-supplied text.

  PRAX-2026-06-15-005  The manual-review threshold and fraud-signal escalation
            are bypassable — business-context heuristics auto-approve
            above-threshold and injection-flagged invoices.
      Action: Make the manual-review threshold and any fraud signal hard,
              non-overridable escalation conditions; remove the
              business-context override branches.

  PRAX-2026-06-15-006  No vendor registration/approval check gates payment —
            and vendor self-registration auto-approves, so any caller can
            create an "approved" vendor and get paid.
      Action: Gate _approve_invoice on vendor.status == 'approved' and remove
              auto-approval from public vendor registration.
