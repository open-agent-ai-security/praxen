BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is *sandbox with the shape of isolation but not the
substance*: the subsystem's entire job is to isolate LLM-generated code from
the host, yet the controls that would deliver that isolation are defaulted off
or absent. LocalCommandLineCodeExecutor substitutes a warnings.warn for the
remit's required production-approval gate and copies the parent process's
complete os.environ — credentials included — into every subprocess, while
create_default_code_executor() silently downgrades Docker to that local
executor on a transient UserWarning. The Docker and docker-Jupyter executors
create containers with no user=, read_only=, mem_limit=, cap_drop=, or network
isolation (docker-Jupyter additionally chmod 0o777s the bind mount and
publishes all ports), so the "standard production path" runs LLM code as root
on the default bridge network. Compounding this, the local executor's own
docstring advertises a dangerous-command regex sanitizer that does not exist
in the code, and no executor writes the per-execution audit log the remit
mandates.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-06-15-001  LocalCommandLineCodeExecutor runs host code behind a
            warnings.warn, not the remit-required production approval gate.
      Action: Add an explicit approval mechanism (e.g. a required confirm
              callback or an env-gated allowlist) that blocks
              LocalCommandLineCodeExecutor execution unless the operator has
              affirmatively approved host execution; a warning is
              insufficient.

  PRAX-2026-06-15-002  Local executor copies the parent process's full
            os.environ — credentials included — into every code subprocess.
      Action: Pass an explicit minimal environment to the subprocess (PATH
              plus only the variables the code legitimately needs) instead of
              os.environ.copy(), or document a required env-allowlist
              parameter and default it to empty.

  PRAX-2026-06-15-003  create_default_code_executor() silently downgrades
            Docker to the local host executor on a transient UserWarning.
      Action: Make the Docker→Local fallback opt-in: raise by default when
              Docker is unavailable, and require an explicit
              allow_local_fallback=True (plus operator approval) before
              returning a host executor.

  PRAX-2026-06-15-004  DockerCommandLineCodeExecutor creates containers with
            no user, read_only, mem_limit, cap_drop, or network isolation —
            root on the default bridge.
      Action: Default the container to user="nobody" (or a non-root UID),
              read_only=True with a tmpfs work area, cap_drop=["ALL"], a
              mem_limit/pids_limit, and network_mode="none" unless egress is
              explicitly configured and approved.

  PRAX-2026-06-15-005  docker_jupyter container runs unhardened as root, chmod
            0o777s the bind mount, and publishes all ports to the host.
      Action 1: Apply the same container hardening as the Docker executor
                (non-root user, cap_drop ALL, read_only, mem/pids limits,
                network_mode none by default) and replace
                publish_all_ports=True with binding only the needed port to
                127.0.0.1.
      Action 2: Replace os.chmod(bind_dir, 0o777) with the minimum permissions
                the kernel user requires.
