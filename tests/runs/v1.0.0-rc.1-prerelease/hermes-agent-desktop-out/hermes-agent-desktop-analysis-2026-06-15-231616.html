Executive Summary
Agent Remit (as declared)
Hermes Agent is a single-tenant personal AI assistant that converses with one operator over a terminal UI and messaging gateways (Telegram, Discord, Slack, Signal, email, SMS, and others), and acts on the operator's behalf through a documented toolset — `terminal`, `code_execution`, `file`, `web`, `browser`, `memory`, `skills`, `delegation`, `cronjob`, `messaging` — while curating its own memory and skills. It must act only for its authorized operator, keep dangerous shell and code behind the OS-level isolation and operator-approval controls its documentation describes, and never exfiltrate operator credentials or session-authorization material outside the trust envelope. The combined remit also covers Hermes Desktop, a non-LLM Electron operator layer that installs, configures, and drives an Agent instance locally, over a plain Remote HTTP connection, or over an SSH tunnel that must bind loopback-only, authenticate key-only as a dedicated non-root user, and pin the remote host key.
Behavior Summary (as observed)

This is a high-maturity, actively-maintained single-tenant agent whose security model is honest about its own limits: its own SECURITY.md states plainly that the only boundary against an adversarial LLM is the operating system, and that the approval gate, output redaction, Skills Guard, and tool allowlists are heuristics, not containment. The dominant risk is the gap between that stated model and the default deployment: the local terminal backend runs LLM-emitted shell and file operations directly on the host, while untrusted content (open web, inbound email, multi-user gateways, operator-configured MCP servers) flows into the same agent context — so a single indirect prompt injection reaches host execution unless the operator has separately stood up Docker/OpenShell whole-process isolation that the code does not require by default.

Layered on that are second-order gaps the team has largely disclosed: dangerous-shell auto-approval fails open in non-interactive/non-gateway/non-cron execution, messaging adapters default to open/pair when no caller allowlist is set, MCP servers launch and their tools execute with no operator-review or approval gate, and the Desktop ships default-on telemetry with no non-root SSH guard and trust-on-first-use host keys. These sit against genuinely strong, operative controls — a fail-closed API-server adapter, an un-overridable credential blocklist (the GHSA-rhgp-j443-p4rf fix), a hardline command floor below YOLO mode, exact-pinned deps with supply-chain CI, and redacting rotating logs — which is why the posture lands at the Established/Partial boundary rather than lower.

Scope of Analysis
Two workspaces analyzed as one agent. The Agent is a large Python codebase (`gateway/`, `tools/`, pluggable terminal backends in `tools/environments/` — `local.py`, `docker.py`, `ssh.py`, `modal.py`, etc.) whose default `local` backend runs `terminal()` shell and the file tools directly on the host; the approval gate (`tools/approval.py`), credential env-scrubbing (`tools/env_passthrough.py`, `tools/environments/local.py`), MCP launch path (`tools/mcp_tool.py` + `tools/osv_check.py`), Skills Guard (`tools/skills_guard.py`), and rotating redacted logging (`hermes_logging.py`) form an in-process control layer the project's own `SECURITY.md` §2 explicitly labels "not a boundary." The Desktop is an Electron/TypeScript app (`src/main/ssh-tunnel.ts`, `ssh-remote.ts`, `ssh-options.ts`, `installer.ts`, `config.ts`, renderer `utils/analytics.ts`); it brokers operator control over SSH (loopback-bound, BatchMode key-only, `accept-new` host-key TOFU) and ships PostHog telemetry default-on in official builds. The repo ships a candid `SECURITY.md`, exact-pinned (`==`) Python dependencies with a hash-locked `uv.lock`, and supply-chain CI (OSV scanner, lockfile check, supply-chain audit).

CRITICAL PRAX-2026-06-15-001 Default `local` terminal backend executes LLM-emitted shell on the host while untrusted web/email/MCP content is ingested with only in-process heuristics in force.
Implement Zero Trust LLM01 — Prompt Injection ASI05 — Unexpected Code Execution (RCE)
Policy Rule — R-15 (Worker Remit):
"Ingesting content from operator-uncontrolled surfaces (open web, inbound email, multi-user channels, untrusted MCP servers) while relying only on in-process heuristics, with no OS-level isolation boundary in force."
tools/environments/local.py:1 — Default terminal backend; runs LLM-emitted shell and file tools (read_file/write_file/patch) directly on the host with no container/sandbox between the model output and the host. SECURITY.md:60 — "The only security boundary against an adversarial LLM is the operating system. Nothing inside the agent process constitutes containment" — §2.2, with §2.4 naming approval/redaction/Skills Guard as heuristics.
Recommended Action
  • Default to a non-`local` terminal backend (or whole-process Docker/OpenShell wrapping) whenever any untrusted input surface (gateway, email, web, MCP) is enabled, and refuse to ingest those surfaces under the `local` backend without an explicit operator break-glass acknowledgement.
  • Surface the chosen isolation posture in the runtime banner/logs so an operator running `local` with untrusted surfaces is warned at startup.
HIGH PRAX-2026-06-15-002 Dangerous-command approval fails open — flagged shell is auto-approved in non-interactive, non-gateway, non-cron execution contexts.
Implement Zero Trust ASI05 — Unexpected Code Execution (RCE) LLM06 — Excessive Agency
Policy Rule — R-12 (Worker Remit):
"Shell execution via the `terminal` tool MUST be subject to the operator approval path before a destructive command runs; auto-approval of shell beyond the operator's configured allowlist is not authorized."
tools/approval.py:983 — "AUTO-APPROVED dangerous command in non-interactive non-gateway context" — logger.warning then `return {"approved": True}` at :988 when not is_cli and not is_gateway (and cron not in deny mode). tools/approval.py:1206 — Mirror path in the tirith/decide flow — `if not is_cli and not is_gateway and not is_ask:` returns approved (:1223) outside cron-deny.

HIGH PRAX-2026-06-15-003 Several messaging adapters fail open — they default to open/pair/allow-all policy when no operator caller allowlist is configured.
Implement Zero Trust ASI03 — Identity and Privilege Abuse ASI09 — Human-Agent Trust Exploitation
Policy Rule — R-01 (Worker Remit):
"Every enabled network-exposed adapter MUST refuse to dispatch agent work, resolve approvals, or relay output until an operator-configured caller allowlist is set; no adapter may fail open when no allowlist is configured."

HIGH PRAX-2026-06-15-004 Operator-configured MCP servers launch automatically and their tools execute with no operator-review or approval gate.
Implement Zero Trust ASI02 — Tool Misuse and Exploitation LLM06 — Excessive Agency
Policy Rule — R-14 (Worker Remit):
"Loading or launching an operator-configured MCP server, which MUST pass the documented supply-chain guard at launch."
tools/mcp_tool.py:3479 — discover_mcp_tools() connects every server gated only by the `enabled` flag — no operator approval/confirm step before launch. tools/mcp_tool.py:3254 — MCP tools auto-register and are exposed/callable; tools/approval.py has no MCP references so exec/delete/send MCP tools run without per-call approval.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Un-overridable provider-credential blocklist (GHSA-rhgp-j443-p4rf fix)

Provider API keys and gateway tokens are stripped by default from shell, MCP, and code-execution children, and a hardcoded blocklist cannot be re-enabled as passthrough by either a malicious skill's frontmatter or operator config.yaml — directly closing the GHSA-rhgp-j443-p4rf sandbox-escape.

tools/env_passthrough.py:48-99 and tools/environments/local.py:126-186

Fail-closed API server adapter authentication

The network-exposed API server adapter refuses to start without an `API_SERVER_KEY`, rejects network-accessible placeholder keys, and uses constant-time bearer comparison returning 401 on mismatch.

gateway/platforms/api_server.py:4146-4167 and :845-866

Hardline command blocklist below YOLO mode

A set of catastrophic commands (`rm -rf /`, `mkfs`, `dd of=/dev/sdX`, fork bomb, shutdown/reboot) is unconditionally blocked before the yolo/smart/mode=off gate, so no session-level setting can bypass it.

tools/approval.py:163-225 and :948-951

Exact-pinned dependencies with supply-chain CI

Python deps are pinned with `==` and a sha256-hash-locked `uv.lock`, enforced by `uv-lockfile-check.yml`, `osv-scanner.yml`, and `supply-chain-audit.yml`.

pyproject.toml:34-48 and .github/workflows/osv-scanner.yml

Redacting rotating logs

All log files use `RotatingFileHandler` with a `RedactingFormatter` so secret-like strings are stripped before reaching disk, across `agent.log`, `errors.log`, and `gateway.log`.

hermes_logging.py:211-298

SSH tunnel is loopback-bound, key-only, fail-closed on changed host key

The Desktop's SSH tunnel forwards to `127.0.0.1` only, uses `BatchMode=yes` (key-only, no password fallback), and surfaces a hard error when the remote host identification changes.

src/main/ssh-tunnel.ts:109-117 and src/main/ssh-remote.ts:111-116
