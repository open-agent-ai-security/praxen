BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is a control plane that is correctly *shaped* for
multi-tenant auth but ships *open* in its default OSS posture: every V1 router
declares an auth dependency via get_dependencies() and annotates itself as
protected by SetAuthCookieMiddleware, yet that middleware is never added in
app.py and the helper returns no dependency unless SESSION_API_KEY is set or
the server runs in SAAS mode — so by default the entire /api/v1 surface,
including the secrets store that holds long-lived git provider tokens, answers
to any unauthenticated caller. This is compounded by a CORS layer that
explicitly falls open to any origin when none are configured and an MCP server
mounted with no auth that exposes PR/MR-creation tools, turning an open API
into a credential-exfiltration and unauthorized-write surface. The genuinely
operative controls sit at the edges — a durable filesystem/SQL event store
gives a structured action record, a JWT-scoped one-shot /webhooks/secrets path
limits token blast radius, and per-session API keys isolate sandbox processes
— but none of them substitute for the missing front-door authentication.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-06-15-001  The entire V1 API, including the secrets store holding
            git provider tokens, is unauthenticated in the default OSS
            posture.
      Action 1: Register an authentication middleware (or fail-closed
                dependency) for /api/v1 by default, so the secrets and git
                routers require a verified caller even in OSS mode rather than
                relying on an unregistered SetAuthCookieMiddleware.
      Action 2: Bind the OSS server to loopback only and document that
                exposing it on a routable interface without SESSION_API_KEY
                publishes the secrets store.

  PRAX-2026-06-15-002  CORS falls fully open to any origin when no origins are
            configured, enabling credentialed cross-origin access to the open
            API.
      Action: Default permitted_cors_origins to localhost only and remove the
              any-origin fallback, or refuse to start with
              allow_credentials=True and an empty origin list.

  PRAX-2026-06-15-003  Unauthenticated open API plus an unguarded MCP
            create_pr/create_mr surface forms a one-hop
            unauthorized-write-and-exfiltration chain.
      Action: Require authentication for the /mcp mount and route
              PR/MR-creation tools through an explicit confirmation gate
              consistent with the remit's "merging/cross-repo writes require
              user confirmation" clauses.
