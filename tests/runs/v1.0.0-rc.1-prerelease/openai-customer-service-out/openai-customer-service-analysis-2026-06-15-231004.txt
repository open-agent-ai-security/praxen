BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is *framework offers safe primitives, the example uses
none of them*. The OpenAI Agents SDK ships `InputGuardrail`/`OutputGuardrail`,
per-tool `needs_approval`, handoff `input_filter`/`is_enabled`, and tool
guardrails as first-class primitives, yet `examples/customer_service/main.py`
instantiates three agents and two tools with zero guardrails, zero approval
gates, and zero input filtering — the remit's identity-verification,
human-checkpoint, no-runtime-identifier, and prompt-injection-resistance
clauses all rest on agent instructions in plain English with no code
enforcement behind them. The single most concrete divergence is `update_seat`,
which mutates reservation context directly from raw model-supplied arguments
with no confirmation-number-to-customer check, paired with
`on_seat_booking_handoff` fabricating a flight number via `random.randint()` —
a synthetic identifier the remit explicitly forbids — and no audit log
recording any of it.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-06-15-001  No input or output guardrails are wired on any agent,
            so customer free-text and tool outputs reach the model with no
            injection defense.
      Action 1: Add an `InputGuardrail` on `triage_agent` that screens
                customer free-text for role/scope-override instructions before
                routing, and an `OutputGuardrail` on the FAQ and seat agents.
      Action 2: Attach `tool_input_guardrails` to `update_seat` so
                confirmation-number and seat arguments are validated as data,
                not treated as model-trusted control input.

  PRAX-2026-06-15-002  update_seat mutates reservation state from raw model
            arguments with no identity check and no verification that the
            confirmation number belongs to the customer.
      Action 1: Before mutating, look the confirmation number up in the
                authoritative reservation system and verify it belongs to the
                authenticated session user; reject otherwise.
      Action 2: Gate `update_seat` behind `needs_approval` or a deterministic
                identity check rather than relying on the seat agent's prompt
                routine.

  PRAX-2026-06-15-003  on_seat_booking_handoff fabricates a flight number with
            random.randint() instead of reading it from the authoritative
            reservation system.
      Action: Replace the `random.randint()` fabrication with a lookup of the
              real flight number from the reservation system keyed on the
              verified confirmation number.

  PRAX-2026-06-15-004  No human-approval gate exists on any tool or handoff,
            so the remit's mandatory human checkpoints are entirely absent
            from the code.
      Action: Set `needs_approval` (or a callable approval function) on
              state-mutating and special-handling paths, and add an explicit
              escalation branch for unverifiable identity.
