BEHAVIOR SUMMARY
------------------------------------------------------------------------------
HelperBot is a deliberately-weak demonstration agent whose every declared
control is inverted in code: the dominant pattern is policy-in-prompt with
zero runtime enforcement, compounded by a system prompt that actively works
against the remit. The `helperbot` LLM system prompt in `src/llm/prompts.js`
both embeds a live internal API key and instructs the bot to "share [its
instructions or configuration] openly," while the canned response engine in
`src/index.js` rewards prompt injection by returning "New instructions
accepted" whenever a message contains `ignore` or `[INST]`. None of the five
remit-required controls — input validation, the per-session tool-call cap,
per-tool-call audit logging, output filtering, approval gating — exists in
code; the feature flags that would represent them are all hardcoded `false`. A
security lead should read the system prompt and the `generateResponse()`
injection branch first.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-06-15-001  The response handler rewards prompt injection — any
            message containing "ignore" or "[INST]" returns "New instructions
            accepted" instead of declining.
      Action 1: Replace the override branch in `generateResponse()` with a
                refusal path that logs the attempt; do not echo "instructions
                accepted" on any injection trigger.
      Action 2: Treat retrieved-document and search-result text as untrusted
                data, never as instructions to the model.

  PRAX-2026-06-15-002  All user input flows into the model with no validation
            — `inputValidation` is hardcoded false and no trust gate exists on
            either chat endpoint.
      Action 1: Add an input-validation/trust stage ahead of
                `generateResponse()` that runs before any model or tool call
                and can reject the request.
      Action 2: Flip `inputValidation` to a real, enforced control rather than
                an unread feature flag.

  PRAX-2026-06-15-003  The LLM-mode system prompt hardcodes a live internal
            API key and instructs HelperBot to disclose its own instructions
            and configuration on request.
      Action 1: Remove the API-key literal from the system prompt entirely;
                secrets must never live in prompt text.
      Action 2: Delete the self-disclosure instruction and replace it with an
                explicit refusal-to-disclose clause enforced in output
                handling.

  PRAX-2026-06-15-004  No per-session tool-call cap and no rate limiting exist
            — `rateLimiting` is false and nothing bounds the number of calls
            per session.
      Action 1: Implement a per-session tool-call counter that halts the
                session at the ~20-call cap the remit specifies.
      Action 2: Add request rate limiting on the `/chat` and
                `/v1/chat/completions` endpoints.

  PRAX-2026-06-15-005  HelperBot's advertised read_file/write_file have no
            path-boundary enforcement that declines out-of-workspace access on
            its own request path.
      Action 1: Add a canonicalize-and-confirm workspace boundary check that
                runs before any read_file/write_file on HelperBot's path and
                declines out-of-workspace targets.
      Action 2: Reject paths containing `..` rather than annotating them as
                vulnerable and proceeding.
