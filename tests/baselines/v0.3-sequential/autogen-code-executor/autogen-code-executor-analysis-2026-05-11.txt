The dominant pattern is sandbox has the shape of isolation but not the
substance, driven by a silent-downgrade chain rooted at
code_executors/__init__.py:58-66 where create_default_code_executor wraps the
Docker construction in a bare except Exception: pass and falls back to the
Local executor on any failure, with only a warnings.warn as the operator
signal. The Local executor it hands back claims in its docstring
(local/__init__.py:57) to sanitize &quot;dangerous commands&quot; via regex
match — no such sanitizer exists in the code path — and at
local/__init__.py:397 the subprocess inherits the parent process's full
os.environ.copy(), directly violating the remit's environment-isolation rule.
The Docker path is no safer than its config allows: containers are created at
docker/_docker_code_executor.py:537-550 with default Linux capabilities, no
--cap-drop, no read-only rootfs, no --security-opt no-new-privileges, no
resource limits, and the caller-supplied extra_volumes dict is merged straight
into the volume mount table (line 546) — a one-line bypass of the remit's
MUST-NOT-mount-host-volumes-without-approval clause. DockerJupyterServer
additionally os.chmod(bind_dir, 0o777) at _jupyter_server.py:328 and ships the
kernel-gateway token over plaintext ws:// (_jupyter_server.py:80, 124-125).
Across all five executors there is zero structured audit logging of
code-execution events; the few logging.* calls cover container lifecycle and
pip-install info, not the per-execution timestamp / source / digest trail the
remit mandates and that Monitor Continuously needs to score above Ad hoc.

CRITICAL FINDINGS

  PRAX-2026-05-11-001  create_default_code_executor silently downgrades from
            Docker to LocalCommandLineCodeExecutor on any Docker failure,
            bypassing the approval requirement for local execution in
            production.
      Action 1: Remove the bare except Exception: pass at
              code_executors/__init__.py:64 and surface Docker construction
              errors to the caller; let the operator application decide
              whether to fall back
      Action 2: Replace the implicit fallback with an explicit
              allow_local_fallback: bool = False kwarg on
              create_default_code_executor; default-deny, and have it raise
              RuntimeError when Docker is unavailable and the flag is not set
      Action 3: When local fallback is allowed, log via logging.warning (not
              warnings.warn) so the event reaches centralized log pipelines,
              and include the original Docker exception in the message

  PRAX-2026-05-11-002  LocalCommandLineCodeExecutor passes the parent
            process's full os.environ to LLM-generated code, leaking every
            caller-side secret into the executed program.
      Action 1: In _execute_code_dont_check_setup, replace env =
              os.environ.copy() with an explicit minimal env dict: PATH
              (rewritten to virtualenv bin if present), HOME (set to a
              per-execution work-dir-scoped path), LANG, and nothing else by
              default
      Action 2: Add an extra_env: Optional[Dict[str, str]] = None constructor
              parameter so the operator can opt specific variables in by name;
              document this as the only supported way to expose caller-side
              state to executed code
      Action 3: Add a regression test under tests/code_executors/ that sets a
              sentinel env var (PRAXA_SECRET_CANARY) in the parent process,
              runs import os; print(os.environ.get('PRAXA_SECRET_CANARY')),
              and asserts the output is empty

  PRAX-2026-05-11-003  DockerCommandLineCodeExecutor accepts caller-supplied
            extra_volumes that are merged directly into the container mount
            table, bypassing work-directory confinement.
      Action 1: Validate every entry of extra_volumes at constructor time:
              reject keys outside an operator-configured allowed_host_paths
              set, reject any path that resolves to /, /etc, /root, the Docker
              socket, or a parent of the work directory, and reject any entry
              whose mode is rw by default
      Action 2: Add an explicit require_volume_approval: bool = True flag;
              when set, any non-empty extra_volumes must be accompanied by an
              extra_volumes_approval_token that the operator application has
              signed
      Action 3: Update the docstring example at
              docker/_docker_code_executor.py:125 to remove host-path mounts
              and replace it with a named-volume example; add a security note
              that mounting host paths defeats the sandbox boundary
