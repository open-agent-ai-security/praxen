Agent Remit (as declared)
The AutoGen Code Execution Subsystem is the execution tier of a two-agent generator/executor pair: it receives LLM-produced code blocks (Python or shell), runs them in a configured environment, and returns stdout, stderr, and exit status to the calling agent. The remit authorizes five executors — LocalCommandLineCodeExecutor, DockerCommandLineCodeExecutor, DockerJupyterCodeExecutor, JupyterCodeExecutor, and AzureContainerCodeExecutor — with the containerized variants positioned as the production path and the Local variant explicitly requiring human approval for production use. Every executor must confine I/O to a configured work directory, enforce a wall-clock timeout, isolate the execution from the parent process's environment and credentials, and record each execution to an audit log carrying timestamp, executor type, source agent, exit status, and a digest of the code.
Agent Structure (as observed)
A Python library packaged in two namespaces: autogen_core.code_executor holds the CodeExecutor abstract base, CodeBlock / CodeResult dataclasses, and the user-defined-function machinery; autogen_ext.code_executors ships the five concrete executors in sibling subpackages (local/, docker/, docker_jupyter/, jupyter/, azure/) plus a shared _common.py with the file-name confinement helper get_file_name_from_content and a silence_pip regex transform. code_executors/__init__.py exposes a create_default_code_executor factory that selects Docker if available and falls back to Local otherwise. Each executor implements its own execute_code_blocks path — direct asyncio.create_subprocess_exec for Local, container.exec_run for Docker, nbclient.NotebookClient for Jupyter, a WebSocket kernel-gateway client for DockerJupyter, and an HTTP POST to code/execute for Azure — none of which share a common policy, approval, or audit layer.
Behavior Summary

The dominant pattern is sandbox has the shape of isolation but not the substance, driven by a silent-downgrade chain rooted at code_executors/__init__.py:58-66 where create_default_code_executor wraps the Docker construction in a bare except Exception: pass and falls back to the Local executor on any failure, with only a warnings.warn as the operator signal. The Local executor it hands back claims in its docstring (local/__init__.py:57) to sanitize &quot;dangerous commands&quot; via regex match — no such sanitizer exists in the code path — and at local/__init__.py:397 the subprocess inherits the parent process's full os.environ.copy(), directly violating the remit's environment-isolation rule.

The Docker path is no safer than its config allows: containers are created at docker/_docker_code_executor.py:537-550 with default Linux capabilities, no --cap-drop, no read-only rootfs, no --security-opt no-new-privileges, no resource limits, and the caller-supplied extra_volumes dict is merged straight into the volume mount table (line 546) — a one-line bypass of the remit's MUST-NOT-mount-host-volumes-without-approval clause. DockerJupyterServer additionally os.chmod(bind_dir, 0o777) at _jupyter_server.py:328 and ships the kernel-gateway token over plaintext ws:// (_jupyter_server.py:80, 124-125). Across all five executors there is zero structured audit logging of code-execution events; the few logging.* calls cover container lifecycle and pip-install info, not the per-execution timestamp / source / digest trail the remit mandates and that Monitor Continuously needs to score above Ad hoc.
