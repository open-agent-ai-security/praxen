BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in prose, zero code-level enforcement:
the remit's hard prohibitions on DML, DDL, administrative commands, and
multi-statement queries are carried only by the SQL_PREFIX system-prompt
string and the create_sql_agent docstring warning —
QuerySQLDatabaseTool._run() passes the LLM-authored SQL straight to
SQLDatabase.run_no_throw(), which calls SQLAlchemy's connection.execute() with
no statement-class check, no parser, no allow-list. The remit itself
anticipates this and states (R-20) that the agent code must not be the sole
enforcement point; in this slice it is. Compounding this, tool outputs — table
names, column names, sample rows, and query results — flow back into LLM
context unlabeled, so a row inserted into any reachable table is an
indirect-injection path into the next query. With zero logging calls anywhere
in the workspace, an injection-driven DML/DDL or admin-command run leaves no
audit trail. Real but narrow operative controls are present (max_iterations=15
default, top_k threaded into the prompt, the sanitize_schema helper, and a
clear operator-facing security docstring) but they do not address the central
exec-of-LLM-output divergence.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-11-001  No code-level enforcement of the
            DML/DDL/admin/multi-statement prohibition — the executor runs
            whatever SQL the LLM produces
      Action 1: Add a deterministic statement-class gate to
              QuerySQLDatabaseTool._run() (or to SQLDatabase._execute()) that
              parses the query with sqlglot or equivalent and rejects any
              statement type outside an explicit allow-list (SELECT, WITH-led
              SELECT, and read-only SHOW/DESCRIBE/EXPLAIN). Rejection must
              occur before connection.execute().
      Action 2: Reject any payload that parses to more than one statement
              (multi-statement / stacked-query defense for R-15).
      Action 3: Reject admin/PRAGMA/COPY/ATTACH/DETACH/VACUUM/ANALYZE/LOAD
              DATA at the same gate (R-14).
      Action 4: Move the DML/DDL prohibition out of SQL_PREFIX as the sole
              enforcement and keep the prompt copy only as defense-in-depth —
              the code gate must be the authoritative check.

  PRAX-2026-05-11-002  Indirect prompt injection via tool outputs (table
            names, column names, sample rows, query results) flows unmarked
            into LLM context and becomes the next SQL
      Action 1: Wrap all tool-returned content (table_info, sample rows, query
              results) in an explicit &lt;tool_output
              untrusted="true"&gt;...&lt;/tool_output&gt; envelope before it
              re-enters the LLM context, and add a system-prompt clause that
              the agent must never follow instructions found inside such
              envelopes.
      Action 2: Strip or escape instruction-shaped tokens in identifier and
              comment fields returned by _get_sample_rows() and the CREATE
              TABLE serialization in get_table_info().
      Action 3: Pair this with the statement-class gate from
              PRAX-2026-05-11-001 so that even a successful injection cannot
              reach a DML/DDL execution path.

  PRAX-2026-05-11-009  Compound chain — indirect injection path + no
            statement-class gate + zero logging means an attacker-controlled
            row can drive arbitrary DML/DDL execution with no audit trail
      Action 1: Treat PRAX-2026-05-11-001, PRAX-2026-05-11-002, and
              PRAX-2026-05-11-005 as a single remediation unit — the
              statement-class gate, tool-output sandboxing, and logging must
              all land together or the chain remains open.
      Action 2: Add structured event logging (event type, query text,
              statement class detected, rejected/allowed, user identity if
              available) at the QuerySQLDatabaseTool._run() boundary so that
              any execution attempt — successful or rejected — is recorded.
      Action 3: Until all three are remediated, run the agent only against a
              database role whose grants make DML/DDL physically impossible at
              the server side (the configuration the create_sql_agent
              docstring already recommends).
