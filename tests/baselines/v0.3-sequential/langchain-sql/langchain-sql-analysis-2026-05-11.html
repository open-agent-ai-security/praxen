LangChain SQL Agent · Analysis completed 2026-05-11

Agent Remit (as declared)
A natural-language interface to a relational database, intended to answer read-only analytical questions for internal data consumers via a configured SQLAlchemy connection. The agent may use sql_db_list_tables, sql_db_schema, sql_db_query_checker, and sql_db_query; output is a direct natural-language reply to the user and nothing else (no outbound HTTP, file writes, email, or caching). DML, DDL, administrative commands, and multi-statement queries are explicitly forbidden with no approval path — and the remit states that the agent code must not be the sole enforcement point for the DML/DDL prohibition.
Agent Structure (as observed)
Python LangChain agent toolkit. create_sql_agent() in libs/community/langchain_community/agent_toolkits/sql/base.py constructs a legacy AgentExecutor wired to four tools defined in libs/community/langchain_community/tools/sql_database/tool.py (QuerySQLDatabaseTool, InfoSQLDatabaseTool, ListSQLDatabaseTool, QuerySQLCheckerTool); all four hold a reference to a shared SQLDatabase wrapper in libs/community/langchain_community/utilities/sql_database.py whose run_no_throw() hands the LLM-authored query string directly to SQLAlchemy via _execute(). There is no SQL parser, no statement-class allow-list, no DML/DDL rejector, no logging in the slice, and no enforced ordering between the checker tool and the executor tool. The system prompt (SQL_PREFIX in agent_toolkits/sql/prompt.py) carries the only operative DML/DDL prohibition; the create_sql_agent docstring (lines 71-97 of base.py) explicitly tells operators that least-privilege roles, statement timeouts, and query guardrails must be applied server-side.
Behavior Summary

The dominant pattern is policy declared in prose, zero code-level enforcement: the remit's hard prohibitions on DML, DDL, administrative commands, and multi-statement queries are carried only by the SQL_PREFIX system-prompt string and the create_sql_agent docstring warning — QuerySQLDatabaseTool._run() passes the LLM-authored SQL straight to SQLDatabase.run_no_throw(), which calls SQLAlchemy's connection.execute() with no statement-class check, no parser, no allow-list. The remit itself anticipates this and states (R-20) that the agent code must not be the sole enforcement point; in this slice it is.

Compounding this, tool outputs — table names, column names, sample rows, and query results — flow back into LLM context unlabeled, so a row inserted into any reachable table is an indirect-injection path into the next query. With zero logging calls anywhere in the workspace, an injection-driven DML/DDL or admin-command run leaves no audit trail. Real but narrow operative controls are present (max_iterations=15 default, top_k threaded into the prompt, the sanitize_schema helper, and a clear operator-facing security docstring) but they do not address the central exec-of-LLM-output divergence.

CRITICAL PRAX-2026-05-11-001 No code-level enforcement of the DML/DDL/admin/multi-statement prohibition — the executor runs whatever SQL the LLM produces
Implement Zero Trust LLM05 — Improper Output Handling ASI02 — Tool Misuse and Exploitation
Policy Rule — R-06, R-12, R-13, R-14, R-15, R-20 (Worker Remit):
"Return only read-only answers sourced from the database / Execute any DML statement ... / Execute any DDL statement ... / Execute administrative or superuser commands ... / Execute multi-statement queries or stored-procedure invocations that could mask hidden DML/DDL / the agent code must not be the sole enforcement point for DML/DDL prohibition — if it is, that is itself a finding"
libs/community/langchain_community/tools/sql_database/tool.py:59 — QuerySQLDatabaseTool._run() returns self.db.run_no_throw(query) — no statement-class check, no parser, the LLM-produced query is passed through verbatim libs/community/langchain_community/utilities/sql_database.py:547 — _execute() runs connection.execute(text(command), parameters, ...) on whatever string the agent supplied; no DML/DDL rejector, no multi-statement detection, no statement allow-list libs/community/langchain_community/agent_toolkits/sql/prompt.py:12 — SQL_PREFIX contains 'DO NOT make any DML statements (INSERT, UPDATE, DELETE, DROP etc.) to the database.' — prompt-only, the only enforcement surface present libs/community/langchain_community/agent_toolkits/sql/base.py:71 — Docstring warning ('!!! warning ... can execute arbitrary SQL against your database ... apply server-side controls: least-privilege roles, statement timeouts, query guardrails') — documents the gap, does not close it
Recommended Action
  • Add a deterministic statement-class gate to QuerySQLDatabaseTool._run() (or to SQLDatabase._execute()) that parses the query with sqlglot or equivalent and rejects any statement type outside an explicit allow-list (SELECT, WITH-led SELECT, and read-only SHOW/DESCRIBE/EXPLAIN). Rejection must occur before connection.execute().
  • Reject any payload that parses to more than one statement (multi-statement / stacked-query defense for R-15).
  • Reject admin/PRAGMA/COPY/ATTACH/DETACH/VACUUM/ANALYZE/LOAD DATA at the same gate (R-14).
  • Move the DML/DDL prohibition out of SQL_PREFIX as the sole enforcement and keep the prompt copy only as defense-in-depth — the code gate must be the authoritative check.

libs/community/langchain_community/utilities/sql_database.py:445 — _get_sample_rows() pulls three rows of raw column values (max_string_length-truncated) and concatenates them into the table_info string returned to the LLM libs/community/langchain_community/utilities/sql_database.py:403 — get_table_info() builds 'table_info' by concatenating CreateTable DDL plus column comments plus sample rows — all raw text from the database, none of it labeled as untrusted in the prompt libs/community/langchain_community/tools/sql_database/tool.py:98 — InfoSQLDatabaseTool._run() returns self.db.get_table_info_no_throw(...) — the result string flows directly back into the LLM as tool output libs/community/langchain_community/agent_toolkits/sql/base.py:201 — Prompt construction for ZERO_SHOT_REACT_DESCRIPTION concatenates prefix + tools + format_instructions + suffix — no separation between trusted system instruction and untrusted tool output / user input
Recommended Action

This is unusual and useful — it sets the right expectations even though it does not close the in-code gap.

Legacy AgentExecutor publicly deprecated with a replacement architecture named

sanitize_schema validator on schema name interpolation
