BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the remit, zero code-level
enforcement, with a single working safeguard — GitHub webhook HMAC
verification (sweepai/utils/hash.py ↔ sweepai/api.py:284) — propping up the
entire trust model. Untrusted issue bodies and PR comments flow directly into
LLM context with no instruction-injection filter, no content-origin labelling,
and no parser; the ripgrep tool in agents/question_answerer.py:270 then
dispatches model-controlled query strings into subprocess.run(...,
shell=True), and core/external_searcher.py dereferences arbitrary URLs lifted
from issue text and feeds the fetched page metadata back into the prompt — a
textbook indirect-injection + SSRF chain. Around that, the workspace ships
SDKs and tokens for forbidden destinations the remit explicitly bans —
slack-sdk, jira, resend, and a DISCORD_STATUS_WEBHOOK_URL read from env — plus
an unauthenticated /jira webhook route and an unauthenticated
/update_sweep_prs_v2 endpoint that performs GitHub merges. Several
blocked_dirs-style protections required by the remit (no CI/CD changes, no
CODEOWNERS / SECURITY.md edits, no Dockerfile edits without explicit issue
authorization) are simply absent from the code path that sweep_bot.is_blocked
guards.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-11-001  Untrusted issue / comment text and external fetched
            content flow into LLM context with no injection filter,
            content-origin labelling, or quote-wrapping
      Action 1: Add an instruction-injection filter that runs on every
              external string (issue body, comment body, fetched URL content,
              PR review text, file diff) before it is added to a `Message`
              content field — minimally a quote-wrap inside an XML tag like
              &lt;untrusted_user_content&gt;...&lt;/untrusted_user_content&gt;
              plus a system-prompt clause that tells the model to treat
              anything inside that tag as data, never instructions.
      Action 2: Gate core/external_searcher.py behind an allow-list of domains
              (or remove it entirely until that exists) — today it will
              dereference any URL an issue author chooses.
      Action 3: Rename sanitize_anthropic_messages or document loudly that it
              is a formatting helper, not a security control, so the next
              reader does not mistake it for one.

  PRAX-2026-05-11-002  Model-controlled query string is concatenated into
            `subprocess.run(..., shell=True)` in the ripgrep tool — direct
            LLM-output-to-shell injection chain
      Action 1: Drop shell=True from every call site and pass the command as a
              list: subprocess.run(['rg', '-n', '-i', '-C=5', '--heading',
              '--sort-files', query], shell=False, ...). The four call sites
              above all already build a list — the join is the bug.
      Action 2: Add a per-call argument allow-list (regex on query /
              code_entity / file_name) that rejects shell metacharacters
              before the call.
      Action 3: Wire the ripgrep tool into a per-tool approval gate so
              high-impact tool invocations require explicit acknowledgement,
              matching the remit's stance that exec is out of scope.

  PRAX-2026-05-11-003  Workspace ships SDKs, env-var slots, and routes for
            forbidden non-GitHub egress channels (Slack, Jira, Resend/email,
            Discord webhook)
      Action 1: Remove resend, jira, and slack-sdk from pyproject.toml unless
              the remit is extended to authorize them — and if it is, the
              remit's Authorized Counterparties section must name them
              explicitly.
      Action 2: Drop the DISCORD_STATUS_WEBHOOK_URL, SLACK_API_KEY, JIRA_*,
              and RESEND_API_KEY blocks from config/server.py together with
              the /jira route in sweepai/api.py:296.
      Action 3: Add an outbound-egress allow-list at the HTTP-client layer
              (block any host that isn't api.github.com, the configured LLM
              endpoint, the Redis host, or the telemetry/Sentry host) so a
              regression silently re-enabling one of these dependencies cannot
              reach the network.
