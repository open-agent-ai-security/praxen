BEHAVIOR SUMMARY
------------------------------------------------------------------------------
OpenHands is a mature, coherently-architected system whose security model is
largely not in this workspace: the controls the Worker Remit cares about most
— sandbox path-confinement, ../symlink rejection, LLM-output-to-tool-call
argument validation, per-session step/time caps, the confirmation gate for
high-impact git/integration actions — live in the separately-versioned agentic
core (openhands-agent-server, the Software Agent SDK), so most behavioral
remit rules are Enforcement-Not-Possible at the layer Praxa can see here
rather than confirmed gaps. The control plane that is here is solid where it
counts: a durable action-level event store, encrypted-at-the-service-layer
secrets, session-scoped sandbox keys with ownership checks, git-branch-name
validation, and a Docker-by-default execution boundary. The dominant pattern
in the visible surface is open-by-default front-door posture in the OSS
configuration: the V1 app server ships with no user authentication when
SESSION_API_KEY/SAAS mode is unset (utils/dependencies.py), a CORS middleware
that returns True for any origin with credentials allowed when
OH_PERMITTED_CORS_ORIGINS is empty (middleware.py), an in-memory
per-request.client.host rate limiter that is ineffective behind a proxy and
resets on restart, and a docker-compose.yml that mounts the Docker socket into
the app container — combined with self-hosted secrets persisted to
~/.openhands/secrets.json in plaintext by FileSecretsStore. None of these is
catastrophic on a single-user localhost box, but together they mean an OSS
operator who exposes the port without reading the docs has a
credential-bearing, container-spawning service open to the network.
