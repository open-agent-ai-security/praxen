Agent Remit (as declared)
OpenHands is an autonomous software-engineering agent that takes natural-language tasks (bug reports, features, issue resolution) and completes them end-to-end — writing code, executing it, browsing the web, and operating connected version-control and issue-tracker accounts (GitHub, GitLab, Bitbucket, Azure DevOps, Forgejo) within the OAuth/PAT scopes the user granted. All agent-generated code must run inside a sandboxed runtime (Docker by default; Kubernetes or a local process runtime in other deployments), never on the host, and file I/O is confined to the per-session sandbox workspace. The task prompt, web content, issue text, PR comments, repository file contents, and microagent definitions are all to be treated as untrusted input; high-impact actions — cross-repo writes, PR merges, force-push/branch-delete/history-rewrite, target-repo dependency changes, CI/CD workflow edits, adding MCP servers, connecting new integrations — require explicit human confirmation, and per-session wall-clock and step-count caps must halt the agent cleanly when exceeded.
Agent Structure (as observed)
The workspace holds OpenHands' control-plane: the V1 FastAPI application server (openhands/app_server/) plus a thin legacy V0 web server (openhands/server/) and a PostHog analytics package (openhands/analytics/). The agentic core — the agent loop, tool implementations, prompt construction, the in-sandbox agent-server that enforces path-confinement and tool-argument validation — has been split into separate pip packages (openhands-agent-server, openhands-aci, the Software Agent SDK) pinned to exact versions in pyproject.toml; it is not present in this workspace. What is here orchestrates sandbox lifecycle (app_server/sandbox/ — Docker by default, with process_sandbox_service.py as a documented host-process fallback for RUNTIME=local/process), a durable per-conversation event store (app_server/event/filesystem_event_service.py, one JSON file per event), encrypted secret storage (app_server/services/jwt_service.py JWE/Fernet over an env- or file-derived key), git-provider integrations, and an MCP server (app_server/mcp/mcp_router.py) that exposes create_pr/create_mr tools and proxies Tavily search. Sandbox-scoped endpoints authenticate with a random 32-byte session API key valid only while the sandbox is RUNNING; the app server itself exposes the V1 REST API behind a CORS middleware that falls open to any origin when no origins are configured, an in-memory per-IP rate limiter (10 req/s), and — in the OSS default — no front-door user authentication. docker-compose.yml bind-mounts /var/run/docker.sock into the app container.
Behavior Summary

OpenHands is a mature, coherently-architected system whose security model is largely not in this workspace: the controls the Worker Remit cares about most — sandbox path-confinement, ../symlink rejection, LLM-output-to-tool-call argument validation, per-session step/time caps, the confirmation gate for high-impact git/integration actions — live in the separately-versioned agentic core (openhands-agent-server, the Software Agent SDK), so most behavioral remit rules are Enforcement-Not-Possible at the layer Praxa can see here rather than confirmed gaps. The control plane that is here is solid where it counts: a durable action-level event store, encrypted-at-the-service-layer secrets, session-scoped sandbox keys with ownership checks, git-branch-name validation, and a Docker-by-default execution boundary.

The dominant pattern in the visible surface is open-by-default front-door posture in the OSS configuration: the V1 app server ships with no user authentication when SESSION_API_KEY/SAAS mode is unset (utils/dependencies.py), a CORS middleware that returns True for any origin with credentials allowed when OH_PERMITTED_CORS_ORIGINS is empty (middleware.py), an in-memory per-request.client.host rate limiter that is ineffective behind a proxy and resets on restart, and a docker-compose.yml that mounts the Docker socket into the app container — combined with self-hosted secrets persisted to ~/.openhands/secrets.json in plaintext by FileSecretsStore. None of these is catastrophic on a single-user localhost box, but together they mean an OSS operator who exposes the port without reading the docs has a credential-bearing, container-spawning service open to the network.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Durable, action-level event log per conversation

Every conversation event is persisted as an individual JSON file under a user-scoped path; the event service supports filtering by kind and timestamp, sorting, and pagination — a real, queryable audit substrate, not a print-statement buffer.

openhands/app_server/event/filesystem_event_service.py:33

Session API keys are random, scoped, and expire with the sandbox

Sandbox-scoped endpoints authenticate with a 32-byte os.urandom key that is rejected once the sandbox leaves RUNNING state, and ownership of the sandbox is checked against the authenticated user before secrets are returned.

openhands/app_server/sandbox/session_auth.py:37

Secrets encrypted at the service layer with algorithm pinning and key rotation

JwtService encrypts persisted values with JWE (A256GCM, alg pinned to 'dir' to block algorithm-confusion) over a key derived from JWT_SECRET or a generated master key, with kid-based rotation and a legacy-Fernet fallback for migration.

openhands/app_server/services/jwt_service.py:233

Git branch names validated before reaching git checkout

ensure_valid_git_branch_name shells out to 'git check-ref-format --branch' to reject malformed/injection-shaped branch names before they are passed to git commands.

openhands/app_server/utils/git.py:24

Provider tokens resolved fresh per request, not cached in long-lived state

AuthUserContext.get_provider_tokens / get_latest_token resolve the current token value at call time from the secrets store rather than holding it in module-level memory, limiting the window for a leaked in-memory token.

openhands/app_server/user/auth_user_context.py:95

Docker sandbox is the default; risky toggles default off

use_host_network and kvm_enabled both default to False, the agent-server image tag is pinned, and the Docker sandbox enforces a max_num_sandboxes cap by pausing the oldest sandboxes — sane secure defaults for the isolation layer.

openhands/app_server/sandbox/docker_sandbox_service.py:638

Security-conscious dependency hygiene with lockfiles and CVE annotations

pyproject.toml carries explicit CVE-driven version constraints (authlib>=1.6.9, orjson>=3.11.6, litellm exclusions) and the repo ships both poetry.lock and uv.lock; the agentic-core packages are pinned to exact versions.
