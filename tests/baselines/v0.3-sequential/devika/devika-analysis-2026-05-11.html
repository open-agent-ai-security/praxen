Agent Remit (as declared)
Devika is positioned as an autonomous software-engineering agent that takes a high-level user request, decomposes it via a planner, researches with a web search API, generates code, and executes it inside a sandboxed runner — all confined to a per-project work directory. The remit authorizes a narrow counterparty list (operator, project workspace, LLM provider, search API, local vector DB) and explicitly forbids execution outside the sandbox, writes outside the work directory, package installs without operator confirmation, exfiltration of credentials or environment variables, and any cross-project memory mixing. High-impact actions — package installs, out-of-bounds writes, breaches of the step or wall-clock cap — are stated to require human approval.
Agent Structure (as observed)
Python Flask + Flask-SocketIO server (devika.py) bound to 0.0.0.0:1337 with a CORS-open Socket.IO instance and an unauthenticated /api/settings route that rewrites API keys and endpoints in config.toml at runtime. A multi-agent orchestrator (src/agents/agent.py — Agent class) dispatches twelve sub-agents (Planner, Researcher, Formatter, Coder, Action, Runner, Feature, Patcher, Reporter, Decision, Answer, InternalMonologue), each rendered from a Jinja2 prompt and routed through src/llm/llm.py to seven pluggable LLM clients (Claude/OpenAI/Gemini/Mistral/Groq/Ollama/LM Studio). The Runner agent's run_code() uses subprocess.run(command_set) directly on LLM-supplied shell strings with no approval gate and a 2-attempt retry loop; Coder.save_code_to_project() joins LLM-supplied filenames to the project dir with no path-traversal check; src/sandbox/code_runner.py and src/sandbox/firejail.py exist as 0-line stubs. Outbound services in code include Netlify deploys and GitHub API calls — neither in the remit's counterparty list.
Behavior Summary

The sandbox the remit relies on is not in the code. src/sandbox/code_runner.py and src/sandbox/firejail.py are both zero-line stubs, while the real execution path — src/agents/runner/runner.py run_code() — calls subprocess.run(command_set) on whatever shell strings the LLM returns, with no allowlist, no per-command approval, and a 2-attempt retry loop that re-prompts the LLM on failure.

This sandbox-shaped-but-empty pattern compounds with three other gaps to form a one-hop external-to-shell chain: an unauthenticated POST /api/settings route in devika.py that rewrites API keys and the OpenAI base URL (allowing model-endpoint hijack), the Researcher fetching web pages via Bing/Google/DuckDuckGo and feeding their raw content through Formatter into the next LLM turn with zero injection filtering, and Coder/Patcher/Feature.save_code_to_project() writing LLM-supplied filenames joined to project_dir with no traversal check. Together: an attacker who can either reach 0.0.0.0:1337 or seed a top web result can redirect the model, write files outside the project dir, and execute arbitrary commands — with no per-project audit log to detect any of it.

What's Working Well

Controls and behaviors that are correctly implemented and verified during this scan. These represent areas where the agent's implementation aligns with its stated policy and security best practices.

Dockerfile drops privileges to a non-root user

The container runs as <code>nonroot</code> (created via <code>useradd -r -g nonroot</code>) with the Devika workdir owned by that user. This is a real (if narrow) blast-radius reduction for the exec-and-traversal findings — host root is not in the immediate reach of a Runner-driven compromise.

devika.dockerfile:33

Per-inference timeout exists and is enforced

<code>LLM.inference()</code> wraps the model call in a <code>ThreadPoolExecutor</code> with a timeout drawn from <code>config.TIMEOUT.INFERENCE</code> (default 60s) and aborts on overrun. It is not the wall-clock cap the remit requires (see PRAX-2026-05-11-013), but it is a real bound and it is wired into the call path.

src/llm/llm.py:127

secure_filename used on project_name in the Project API blueprint

<code>src/apis/project.py</code> applies <code>werkzeug.utils.secure_filename</code> to <code>project_name</code> on create/delete/download routes — partial defense against trivial path bugs in those routes. The same hygiene is missing elsewhere (cf. PRAX-2026-05-11-002, PRAX-2026-05-11-007), but the pattern is correct where present.

src/apis/project.py:21

Settings response body is redacted from the HTTP access log

<code>route_logger</code> replaces the response body with "*** Settings are not logged ***" when the request path contains 'settings'. The in-memory and on-disk credential exposures are separate and worse (see PRAX-2026-05-11-005), but this single specific redaction is correctly implemented.
