BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is framework offers safe primitives, example uses none of
them. The OpenAI Agents SDK ships InputGuardrail / OutputGuardrail, per-tool
needs_approval, ToolInputGuardrail / ToolOutputGuardrail, an identity-scoped
RunContextWrapper, and structured tracing — yet
examples/customer_service/main.py declares zero guardrails on any of the three
agents and leaves needs_approval=False on update_seat, the only
reservation-mutating tool. Every safety-relevant remit rule — identity
verification before mutation, refusal of refund/rebooking/upgrade requests,
rejection of instructions embedded in customer free-text or tool outputs,
audit logging of seat changes — currently lives only in the natural-language
instructions= strings of the FAQ and Seat Booking agents. That is the textbook
prompt-only enforcement pattern: a single jailbreak collapses every control.
The one operative SDK default the agent inherits is workflow tracing
(RunConfig.tracing_disabled=False), which provides observability but is not
the per-mutation audit log the remit calls for. A focused remediation — add an
InputGuardrail to Triage, set needs_approval=True on update_seat, and add a
verification check at the top of update_seat that the confirmation_number
belongs to the authenticated session — closes most of the gap without leaving
the SDK.
