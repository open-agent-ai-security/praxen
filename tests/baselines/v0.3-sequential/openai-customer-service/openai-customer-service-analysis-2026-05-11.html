OpenAI Customer Service Agent · Analysis completed 2026-05-11

Agent Remit (as declared)
A multi-agent customer-service system for an airline. A Triage Agent routes each customer turn to one of two specialists — a FAQ Agent that answers questions from a curated FAQ dataset using faq_lookup_tool, and a Seat Booking Agent that updates seat assignments on existing confirmed reservations via update_seat. The remit forbids any change to fares, payments, refunds, or rebooking; it requires identity verification (passenger name + confirmation number) before any reservation mutation, an audit log of every seat change, and outright rejection of instructions embedded in free-text customer input or in tool outputs.
Agent Structure (as observed)
Python application built on the OpenAI Agents SDK (openai-agents v0.17.1, MIT). The example lives in examples/customer_service/main.py: three Agent[AirlineAgentContext] instances (Triage, FAQ, Seat Booking) wired together with bi-directional handoff() calls and an on_seat_booking_handoff hook that generates a flight number with random.randint(100, 999). Tools are two @function_tool-decorated coroutines (faq_lookup_tool, update_seat); no input or output guardrails are attached to any agent, needs_approval is at the SDK default (False) on every tool, and identity is never verified — AirlineAgentContext defines a passenger_name field but the example never sets or checks it. The SDK's tracing layer is implicitly active (RunConfig.tracing_disabled defaults to False) and a trace("Customer service", group_id=conversation_id) block wraps the run loop.
Behavior Summary

The dominant pattern is framework offers safe primitives, example uses none of them. The OpenAI Agents SDK ships InputGuardrail / OutputGuardrail, per-tool needs_approval, ToolInputGuardrail / ToolOutputGuardrail, an identity-scoped RunContextWrapper, and structured tracing — yet examples/customer_service/main.py declares zero guardrails on any of the three agents and leaves needs_approval=False on update_seat, the only reservation-mutating tool.

Every safety-relevant remit rule — identity verification before mutation, refusal of refund/rebooking/upgrade requests, rejection of instructions embedded in customer free-text or tool outputs, audit logging of seat changes — currently lives only in the natural-language instructions= strings of the FAQ and Seat Booking agents. That is the textbook prompt-only enforcement pattern: a single jailbreak collapses every control. The one operative SDK default the agent inherits is workflow tracing (RunConfig.tracing_disabled=False), which provides observability but is not the per-mutation audit log the remit calls for. A focused remediation — add an InputGuardrail to Triage, set needs_approval=True on update_seat, and add a verification check at the top of update_seat that the confirmation_number belongs to the authenticated session — closes most of the gap without leaving the SDK.

CRITICAL PRAX-2026-05-11-001 update_seat mutates reservation state with no identity verification and no human approval — the SDK's needs_approval gate is at default False and AirlineAgentContext.passenger_name is never populated or checked.
Implement Zero Trust LLM06 — Excessive Agency ASI03 — Identity and Privilege Abuse
Policy Rule — R-01, R-02, R-06, R-11, R-12 (Worker Remit):
"Confirm the customer's identity (passenger name + confirmation number) before any action that mutates reservation state / Verify that a submitted confirmation number belongs to the authenticated customer before reading or modifying that reservation / Modify reservations or seat assignments for a customer other than the authenticated session user / Hand off between agents in a way that escalates privileges or bypasses identity verification / Issue a seat change without first verifying the confirmation number against the authoritative reservation record"
examples/customer_service/main.py:67 — @function_tool on update_seat with no needs_approval=True — SDK default is False per src/agents/tool.py:328-335 examples/customer_service/main.py:79 — context.context.confirmation_number = confirmation_number — writes model-supplied confirmation number into context with no check against the authenticated user examples/customer_service/main.py:30 — AirlineAgentContext.passenger_name is declared but never populated, read, or compared anywhere in the example examples/customer_service/main.py:109 — seat_booking_agent = Agent(...) — no input_guardrails, no output_guardrails, no tool_input_guardrails on update_seat
Recommended Action
  • In examples/customer_service/main.py, set needs_approval=True on the update_seat tool (or wire a ShellApprovalFunction-shaped callback that checks the session principal) so seat mutations require RunState.approve().

  • Attach an InputGuardrail on the Triage Agent and Seat Booking Agent that requires the authenticated principal to be present in the run context before any handoff to Seat Booking proceeds.

Recommended Action

  • Add a ToolOutputGuardrail on faq_lookup_tool that strips or refuses tool outputs containing imperative instructions to the model.
