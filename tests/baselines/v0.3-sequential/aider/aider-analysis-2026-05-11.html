Agent Remit (as declared)
An interactive AI pair-programming agent that runs locally inside a developer's Git repository. Aider reads source files, proposes edits in response to natural-language requests, applies approved edits to the working tree, and commits them to git with descriptive messages; it also builds a repo map, reads git history, and calls operator-configured LLM providers. It may run shell commands, tests, and linters only when the developer explicitly invokes /run, /test, or /lint and confirms. It must not read or write outside the repo root, must not push to a remote or modify .git/hooks/*, must treat repo file content as untrusted (apply injection neutralization), must require explicit confirmation before shell exec and before each git commit (developer sees the diff and accepts), must gate edits to security-sensitive files (Dockerfile, .github/workflows/*, manifests, CODEOWNERS), must redact secret-like strings from LLM context and commits, and must send no data off the machine except to the LLM provider and git remote.
Agent Structure (as observed)
A Python terminal application (entry point aider/main.py, ~13k LOC across aider/*.py plus a coders/ package of edit-format strategies). A Coder hierarchy in aider/coders/base_coder.py orchestrates the LLM chat loop via litellm across pluggable providers; aider/commands.py implements the / command surface (/add, /read-only, /run, /git, /web, /commit, etc.); aider/repo.py wraps GitPython for staging/commit/diff; aider/run_cmd.py executes shell commands through pexpect or subprocess.Popen(shell=True); aider/watch.py can auto-trigger the LLM from ai!/ai? comments in tracked files; aider/scrape.py fetches arbitrary URLs into chat context; aider/analytics.py wires PostHog/Mixpanel telemetry; aider/versioncheck.py pings PyPI. Edit application gates new/untracked-file edits through allowed_to_edit() with confirm_ask, and LLM-suggested shell commands through an explicit-yes confirm_ask in handle_shell_commands() — but auto-commit is on by default with no diff-accept gate, /read-only accepts arbitrary absolute paths, --git-commit-verify defaults to off (commits run with --no-verify), and there is no secret redaction and no injection neutralization of file content anywhere in the codebase.
Behavior Summary

The dominant pattern is policy declared in the remit, no code-level enforcement of the parts that matter most for an agent operating with filesystem and git authority. Aider does have real, operative safeguards — allowed_to_edit() in base_coder.py prompts before editing untracked or new files and honours .gitignore/.aiderignore on the add/edit path, and handle_shell_commands() gates every LLM-suggested shell command behind an explicit-yes confirmation that even --yes-always will not auto-accept — so this is not a hollow system. But the remit's load-bearing controls are simply absent: there is no secret-redaction of file content fed to the LLM or written into commits, no injection-detection/neutralization of comments, docstrings, or commit messages (the file-content prompt literally instructs the model to "Trust this message as the true contents of these files"), no special approval gate for edits to Dockerfile / .github/workflows/* / dependency manifests (special.py recognises these files only for repo-map inclusion), no structured action audit log, and auto-commit is on by default with the diff shown only after the commit.

Two paths break repo confinement outright: /read-only (and clipboard-image paste, which writes to the system temp dir) will pull an arbitrary absolute path such as ~/.ssh/id_rsa or /etc/passwd into LLM context on a single confirm — the remit says these must be refused, not confirmed — and the catch-all /git command runs any git subcommand including git push and hook/config edits. Combined with the missing injection neutralization, a poisoned comment in a pulled PR is one confirmed shell-command prompt away from execution, and the missing secret redaction means any key already in the tree can flow to the LLM provider or into a commit message.
