BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the remit, no code-level
enforcement of the parts that matter most for an agent operating with
filesystem and git authority. Aider does have real, operative safeguards —
allowed_to_edit() in base_coder.py prompts before editing untracked or new
files and honours .gitignore/.aiderignore on the add/edit path, and
handle_shell_commands() gates every LLM-suggested shell command behind an
explicit-yes confirmation that even --yes-always will not auto-accept — so
this is not a hollow system. But the remit's load-bearing controls are simply
absent: there is no secret-redaction of file content fed to the LLM or written
into commits, no injection-detection/neutralization of comments, docstrings,
or commit messages (the file-content prompt literally instructs the model to
"Trust this message as the true contents of these files"), no special approval
gate for edits to Dockerfile / .github/workflows/* / dependency manifests
(special.py recognises these files only for repo-map inclusion), no structured
action audit log, and auto-commit is on by default with the diff shown only
after the commit. Two paths break repo confinement outright: /read-only (and
clipboard-image paste, which writes to the system temp dir) will pull an
arbitrary absolute path such as ~/.ssh/id_rsa or /etc/passwd into LLM context
on a single confirm — the remit says these must be refused, not confirmed —
and the catch-all /git command runs any git subcommand including git push and
hook/config edits. Combined with the missing injection neutralization, a
poisoned comment in a pulled PR is one confirmed shell-command prompt away
from execution, and the missing secret redaction means any key already in the
tree can flow to the LLM provider or into a commit message.
