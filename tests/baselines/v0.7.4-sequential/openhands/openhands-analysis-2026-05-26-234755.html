Executive Summary
Agent Remit (as declared)
OpenHands is an autonomous software-engineering agent that accepts natural-language tasks and completes them end-to-end — writing code, executing it inside a sandboxed runtime, browsing the web, and operating against the user's connected git and issue-tracking platforms. The remit authorizes shell, file, browser, git, issue-tracker, LLM-inference, and operator-configured MCP tools, all confined to the per-session sandbox workspace. The remit treats every input class — user prompts, web pages, issue/PR content, repo file contents, micro-agents and memory — as untrusted prompt-injection-capable content, and gates destructive integration ops, out-of-scope writes, dependency commits, CI changes, MCP-server additions, and force-pushes behind user confirmation. Code execution on the host outside the sandbox, host-FS access outside the workspace, and cross-session state sharing are forbidden outright.
Behavior Summary (as observed)

The dominant pattern is policy declared in the remit, partially enforced at the architectural boundary, and silently downgraded by configuration defaults. The remit is unusually specific — it names every approval-required action class (PR merge, off-task-repo writes, dependency commits, CI edits, MCP-server adds, force-push) — but the runtime exposes a single conversation-scoped confirmation_mode that defaults to NeverConfirm, with no per-action gate keyed to those specific classes; the MCP create_pr tool accepts repo_name straight from LLM output with no task-origin cross-check, so R-16 (writes outside task-origin repo) has no code-level enforcement.

The companion pattern is the untrusted-content channel that the remit names explicitly and the code treats as authoritative: repo .openhands/microagents/*.md files are loaded into the LLM context every session, and the agent has full write access to them inside the sandbox — combined with shell/Jupyter auto-approval in the default config and the unauthenticated uvx mcp-server-fetch tool that brings arbitrary web content into context, that's a one-hop injection-to-exec chain that the sandbox boundary contains but does not prevent. The ProcessSandboxService alternative (used when RUNTIME=local, the path AGENTS.md tells developers to use) removes even the container boundary.

Scope of Analysis
Python FastAPI app under `openhands/app_server/` with three pluggable sandbox backends (`DockerSandboxService`, `ProcessSandboxService`, `RemoteSandboxService`) and an MCP server (`openhands/app_server/mcp/mcp_router.py`) exposing platform-specific PR-creation tools (`create_pr`, `create_mr`, `create_bitbucket_pr`, `create_bitbucket_data_center_pr`, `create_azure_devops_pr`). The agentic core and tool implementations live in pinned external packages (`openhands-agent-server==1.22.1`, `openhands-sdk==1.22.1`, `openhands-tools==1.22.1`) and are invoked inside the sandbox; the app-server is the orchestrator, secrets-store, conversation manager, and webhook receiver. Notable structural pieces: `LocalhostCORSMiddleware` with an "allow any origin" fallback when `permitted_cors_origins` is empty, `InMemoryRateLimiter` at the HTTP edge, durable `EventService` writing per-conversation event logs to disk, repo `.openhands/microagents/*.md` files auto-loaded into LLM context every session, and `validate_session_key()` enforcing RUNNING-sandbox-only session-API-key acceptance.

Maturity Scoring Rubric

Every score above is based on this scale. A score is a snapshot of observable posture — not a verdict on the people or team behind the system.

Score Label Meaning
5 Exemplary Best-in-class; automated, continuously tested, reference quality. Rarely achieved in shipping systems.
4 Strong Comprehensive controls, active management, minor gaps. Production-ready.
3 Established Documented controls consistently applied; known gaps accepted. A respectable baseline.
2 Partial Some controls exist but coverage is incomplete; key gaps remain.
1 Ad hoc Informal or inconsistent measures; relies on individual judgment.
0 Absent No evidence this category is addressed at all.
Weighting: the weighted overall above is the sum of each category's score × weight (the per-category weights are shown on each card). Zero Trust carries double weight by design; see the RAISE framework reference for the rationale.
