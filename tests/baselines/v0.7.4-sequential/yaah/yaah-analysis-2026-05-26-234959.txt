BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The single most important pattern is **uniform protections promised,
non-uniform protections shipped**: `pkg/generator/hookmap.go` declares no
`PreToolUse` or `PostToolUse` mapping for the Codex CLI target, and
`pkg/generator/codex.go:GenerateHooks` skips any event whose Codex name is
empty — so a developer who runs `yaah generate --agent codex` receives a
`.codex/hooks.json` wired only for `SessionStart`/`Stop`, with the command
guard, secret scanner, linter, and comment checker silently absent from that
agent's runtime path. The same fragility appears one layer deeper in `npx -y
@context7/mcp` in `pkg/mcp/providers/context7.go` (and
`@notionhq/notion-mcp-server` in `notion.go`): the Worker Remit says
third-party MCP servers MUST be pinned to a known-good, integrity-checked
version, but the configured invocation auto-installs the latest publish from
npm on every run with no integrity check. A third structural gap — the remit's
"durable, structured, timestamped record of tool calls, blocked actions, and
file modifications" — is half-met: `pkg/harness/harness.go:HandleHookEvent`
does record tool calls and blocks into a per-session JSON store, but
`pkg/hooks/handlers/sessionlog.go` only logs
`SessionStart`/`SessionEnd`/`Stop` events to a plaintext line in
`.claude/logs/sessions.log`, so on the Codex target (where the JSON store is
never populated for tool calls) the durable audit trail required by the remit
does not exist.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-26-001  Codex-target generated configuration ships without
            command guard, secret scanner, linter, or comment checker because
            no PreToolUse / PostToolUse mapping exists.
      Action 1: In `pkg/generator/hookmap.go`, add the Codex mappings for
                `HookPreToolUse` and `HookPostToolUse` (Codex CLI does expose
                these hook events) so the command guard and secret scanner
                reach the Codex runtime path.
      Action 2: Add a generator-level invariant test that asserts each
                target's generated hook config carries `command-guard` on the
                equivalent of `PreToolUse` and `secret-scanner` on
                `PostToolUse`, failing the build if any target drops a
                declared protective hook.

  PRAX-2026-05-26-002  Third-party MCP servers (context7, notion) are launched
            with `npx -y` and no version pin, contradicting the remit's
            pinning rule.
      Action 1: Pin both providers to a known-good npm version (e.g.
                `@context7/mcp@1.2.3`) and document the version-bump review
                process; better, fetch the package once into a checked-in
                lockfile or vendor directory and invoke from there.
      Action 2: Add a startup integrity check in `pkg/mcp/registry.go` that
                refuses to register an MCP provider whose `Command` is `npx`
                with `-y` and no `@` suffix, with an explicit override flag
                for development.
