BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is framework offers safe primitives, example wires none
of them in: the agents SDK ships guardrail.py and tool_guardrails.py alongside
this example, but main.py attaches neither input nor output guardrails to any
of the three agents and relies entirely on natural-language instructions in
the agent prompts to enforce scope, identity, and tool use. The most
exploitable concrete consequence is the update_seat tool: the remit requires
that the customer's identity be confirmed and the confirmation number be
verified against the authoritative reservation record before any seat
mutation, but the implementation accepts both the confirmation number and the
new seat as raw model-supplied strings, writes them straight into
context.context, and never calls a reservation system at all — an
authenticated-customer model is assumed but never enforced in code, so any
prompt that reaches the seat-booking agent gets seat-state writes for free.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-26-001  Seat-booking agent mutates reservation state with no
            customer-identity confirmation in code.
      Action 1: Add an explicit authenticated_customer_id field to
                AirlineAgentContext populated from the session before any
                agent runs, and have update_seat reject the call when it is
                None.
      Action 2: Wire an input_guardrails entry on seat_booking_agent that
                fails the turn when the context lacks a verified customer
                identity, using the SDK's guardrail.py primitives the
                framework already ships.

  PRAX-2026-05-26-002  Confirmation-number ownership is never checked against
            the authoritative reservation system before seat mutation.
      Action: Replace the in-context assignment with a call to a real
              reservation-lookup function that takes the authenticated
              customer id and the submitted confirmation number, and returns
              the reservation only when ownership matches; reject the tool
              call when no match is returned.

  PRAX-2026-05-26-008  Compound chain — untrusted user input flows directly to
            an unauthenticated write tool with no audit, allowing one-prompt
            seat hijack of any reservation.
      Action: Fix the three contributing findings together — input guardrails
              on triage, an authenticated-customer + reservation-lookup gate
              inside update_seat, and a durable audit-log write — rather than
              patching them individually; until all three are in place this
              chain remains live even if any two are mitigated.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-26-003  Seat availability and existence are never validated
            before update_seat returns success.
      Action: Look up the flight's seat map by confirmation number, reject
              seats that are not in the map or are already assigned to another
              passenger, and return a structured error to the agent so the LLM
              can ask for an alternative.

  PRAX-2026-05-26-004  Durable seat-change audit log mandated by the remit is
            not implemented.
      Action: Add an append-only audit writer (database table or JSONL file)
              called from inside update_seat with the six required fields
              before returning success.

  PRAX-2026-05-26-005  FAQ agent's no-speculation rule is enforced only by
            prompt text, not by code.
      Action: Attach an output_guardrails entry to faq_agent that fails the
              turn when the assistant message was emitted without a preceding
              faq_lookup_tool call (or when the message text is not a
              substring of the tool's last return value).

  PRAX-2026-05-26-006  No detection or handling of injection in customer input
            or tool outputs is wired into the agents.
      Action 1: Attach input_guardrails on triage_agent that flag
                override-style instructions in the user message (e.g. "ignore
                previous", "you are now") and either drop the turn or escalate
                to a human checkpoint.
      Action 2: Attach tool_guardrails that treat the faq_lookup_tool and
                update_seat outputs as opaque data — never as instructions —
                and prevent them from influencing handoff or tool-selection
                decisions.

  PRAX-2026-05-26-007  Flight number is fabricated at runtime via
            random.randint instead of read from the reservation system.
      Action: Replace the random flight-number generation in
              on_seat_booking_handoff with a lookup against the reservation
              system using the confirmation number the customer provides;
              refuse to proceed when no reservation is found.
