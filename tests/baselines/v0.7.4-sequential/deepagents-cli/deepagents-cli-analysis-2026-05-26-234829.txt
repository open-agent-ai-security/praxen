BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The CLI's central guarantee — *only what the developer declared, nothing more*
— is well-implemented at the seam between project layout and bundle artifact:
stdio MCP is rejected pre-deploy, `.env` is explicitly excluded from the
seeded-memory payload, the deploy bundle's dependency set is derived only from
declared providers, and an anonymous-auth deploy hits a colour-coded warning
and an explicit `[y/N]` confirmation. The gaps cluster on the other side of
that seam — what gets *taken on faith* once a project is "declared." Remote
MCP servers in the user's `mcp.json` are copied verbatim into the deployment
with no version pin or integrity check (the remit's Known-Good-Baseline rule
explicitly says they MUST be pinned), the http/sse transport check accepts
plain `http://` URLs without warning, and the CLI's own threat model — the
artifact the remit's *Escalation and Limits* clause asks to be kept current —
carries a disclaimer that whole sections of it no longer apply after the 0.1.0
package split. None of these is a runtime injection chain on its own, but
together they describe a posture where the *bundling* path is tightly held and
the *trust posture of the deployment* leans heavily on what the developer
wrote in their `mcp.json` and whether they happen to read the colour-coded
prompt.
