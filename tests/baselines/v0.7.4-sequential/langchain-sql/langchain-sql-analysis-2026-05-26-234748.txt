BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in prompt, zero code-level
enforcement: the remit's hard prohibitions on DML, DDL, administrative
statements, and multi-statement queries are recited inside the `SQL_PREFIX`
system prompt, but the execution path is `LLM output →
QuerySQLDatabaseTool._run → db.run_no_throw → _execute →
engine.begin().execute(text(query))` with no SQL parser, statement-type
filter, table-scope check, or human approval step between the model and the
database. The `sql_db_query_checker` tool reads as a safety gate but is only
an LLM call about SQL correctness (NULL-IN, UNION ALL, type casts) — it cannot
decline a syntactically valid `DROP TABLE`. The remit's own defense-in-depth
clause (R-24) explicitly forbids the agent's logic being the sole enforcement
point — and the agent's logic is the only enforcement attempted, since logging
and audit are absent. Operators who deploy with a writable DB role inherit a
one-prompt-injection-away path from a user question to row deletion with no
audit trail.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-26-001  Compound chain — user question reaches the database
            through an unfiltered, unaudited execution path because the
            remit's prohibitions live only in the system prompt.
      Action 1: In `QuerySQLDatabaseTool._run`
                (tools/sql_database/tool.py:53-59), parse the incoming query
                with `sqlglot` or equivalent and reject any statement whose
                type is not `SELECT`, `WITH`, `SHOW`, `EXPLAIN`, or `DESCRIBE`
                — return an error string rather than calling
                `db.run_no_throw`; emit a structured log line with the
                rejected SQL and the agent run id regardless of outcome.
      Action 2: Document in `create_sql_agent`'s docstring that the read-only
                DB role is mandatory (not advisory) for any non-toy
                deployment, and add a constructor parameter
                (`require_readonly_role=True`) that probes the role's grants
                at construction and raises if write permissions are present.

  PRAX-2026-05-26-002  Data-modifying statements (INSERT, UPDATE, DELETE,
            MERGE, REPLACE) execute through `sql_db_query` with no code-side
            filter — the remit's read-only requirement is unenforced at the
            agent layer.
      Action 1: In `QuerySQLDatabaseTool._run`, parse the query (e.g.,
                `sqlglot.parse_one(query).find(exp.Insert | exp.Update |
                exp.Delete | exp.Merge)`) and return `"Error: data-modifying
                statements are not permitted by this agent — only SELECT,
                WITH, SHOW, EXPLAIN, and DESCRIBE are allowed."` for any
                non-read statement.
      Action 2: Add unit tests under `tests/unit_tests/agents/test_sql.py`
                that drive the agent with prompts asking it to
                INSERT/UPDATE/DELETE and assert the tool returns the rejection
                string rather than executing.

  PRAX-2026-05-26-003  DDL, administrative, and multi-statement SQL execute
            through `sql_db_query` with no parser or filter — three remit
            prohibitions (R-08, R-09, R-10) are unenforced at the agent layer.
      Action 1: Extend the parser-based filter in PRAX-2026-05-26-002's
                recommendation to also reject any AST containing `exp.Drop`,
                `exp.AlterTable`, `exp.Create`, `exp.TruncateTable`,
                `exp.Grant`, `exp.Revoke`, `exp.AttachDatabase`, `exp.Command`
                (admin), and any parse result whose `statements` list has
                length > 1.
      Action 2: Add a `tests/unit_tests/agents/test_sql_safety.py` suite that
                fuzzes the agent with prompts asking for each forbidden
                statement class (DROP TABLE, ALTER TABLE, GRANT ALL,
                multi-statement payload separated by `;`) and asserts
                rejection at the tool layer.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-26-004  `sql_db_query_checker` is an LLM correctness review
            (NULL handling, joins, type casts) — not a safety gate; nothing in
            code requires it before `sql_db_query`.
      Action 1: Either reframe `sql_db_query_checker` as a true safety gate
                (extend `QUERY_CHECKER` to require refusal of any non-SELECT
                statement and have the checker return a structured `{safe:
                bool, reason: str}` rather than rewritten SQL) and make
                `QuerySQLDatabaseTool._run` require a recent safe-check token,
                or remove the rule R-04 as enforceable-at-this-layer and rely
                on the parser filter from PRAX-2026-05-26-002.
      Action 2: Update the toolkit's `query_sql_database_tool_description`
                (toolkit.py:108-115) to stop telling the model the checker is
                optional and to instead state that the executor is the
                authoritative gate (after the parser filter lands).

  PRAX-2026-05-26-005  Returned row content, table names, and column names
            re-enter the LLM context without sanitization or
            injection-detection — R-13 and R-20 are unenforced.
      Action 1: Add an `_audit_returned_content` step in `SQLDatabase.run`
                that scans each row's string fields for known injection
                prefixes (e.g., `"Ignore previous instructions"`, leading
                `system:` / `assistant:` markers, fenced tool-call JSON) and
                either replaces matches with a `[REDACTED — returned-content
                injection candidate]` marker or annotates the row with a
                sentinel the agent prompt is taught to treat as untrusted.
      Action 2: In `SQL_PREFIX`, add an explicit sentinel-aware instruction
                telling the model that any sentinel-marked row content is to
                be reported to the user as suspicious data rather than acted
                on.

  PRAX-2026-05-26-006  Zero logging in the SQL Agent code path — no record of
            agent inputs, tool invocations, generated SQL, or rejected
            statements.
      Action 1: Add a module-level `logger =
                logging.getLogger("langchain_community.sql_agent")` to
                `tools/sql_database/tool.py` and emit structured INFO events
                at the start of every `_run` (with the tool name, the
                agent-run id from `run_manager`, and the query/parameters) and
                on exception (with `exc_info=True`).
      Action 2: In `SQLDatabase.run`, emit an INFO event before `_execute`
                with the parsed statement type (after the parser filter from
                PRAX-2026-05-26-002 lands) and a WARNING event whenever the
                filter rejects a query — so operators can monitor for
                rejection volume as a signal of injection attempts in flight.

  PRAX-2026-05-26-007  All defense against role/instruction override (R-12) is
            prompt-only — there is no separate code-side defense against user
            text rewriting the agent's effective policy.
      Action 1: Before the user `input` reaches the prompt, run it through an
                injection-detection step (e.g., a small classifier or a
                deny-list of role-confusion markers like `system:`,
                `assistant:`, ``, `ignore previous`) and either reject or
                sentinel-mark suspicious questions; emit a WARNING log line on
                detection so operators can monitor volume.
      Action 2: Reframe the agent so the policy-critical executor-side parser
                filter (PRAX-2026-05-26-002, -003) is the load-bearing defense
                and the prompt is documentation, not enforcement.
