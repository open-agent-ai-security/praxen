BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is *framework offers safe primitives, code/example uses
none of them, and several primitives carry the shape of isolation but not the
substance*. The remit declares six mandatory controls every executor MUST
always do (work-directory confinement, wall-clock timeout, complete stdio
capture, host-caller isolation, scoped persistence, and an audit log per
execution); only the first three are reliably implemented across the executor
set. The audit-log requirement — timestamp, executor kind, agent, work_dir,
timeout, exit, digest — is entirely absent across every executor; the few
`logging.info`/`logging.debug` calls present cover container lifecycle and pip
installs, not executions. The host-isolation requirement is broken in two
ways: the local executor passes the entire host environment to the child via
`os.environ.copy()` in `local/__init__.py`, and the Docker executor accepts
caller-supplied `extra_volumes` and `extra_hosts` and runs containers as the
image default (typically root) with no network policy, leaving the container
as a thin namespace boundary rather than a real isolation layer. Compounding
this, `create_default_code_executor` will silently fall back from Docker to
the local executor when Docker is unavailable — bypassing the remit's "local
execution requires human approval" clause with only a `UserWarning` — and the
local executor's own docstring claims "Command line code is sanitized using
regular expression match against a list of dangerous commands" while no such
code exists in the file.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-26-001  No executor records a per-execution audit log, in
            direct contradiction of the remit's mandatory audit-log invariant.
      Action 1: Add an audit emitter on the abstract `CodeExecutor` base class
                (`python/packages/autogen-core/src/autogen_core/code_executor/_base.py`)
                that every concrete executor must call exactly once per code
                block with `{timestamp, executor_kind, language, source_agent,
                work_dir, timeout, exit_code, code_sha256}` — emit before and
                after execution so timeouts and exceptions are still recorded.
      Action 2: Wire the emitter through `PythonCodeExecutionTool.run` so the
                AutoGen tool layer can capture `source_agent` from the calling
                agent context that the executor cannot see directly.

  PRAX-2026-05-26-002  Local executor copies the full host environment into
            every child process, leaking credentials and tokens to
            LLM-generated code.
      Action: Replace `os.environ.copy()` with an explicit allow-list of
              environment variables to forward (default: `PATH` plus an
              optional caller-supplied list); document the change in the class
              docstring and add a regression test asserting that an env var
              set on the parent process is not visible in the child's
              `os.environ`.

  PRAX-2026-05-26-003  Docker executor creates containers as the image default
            user (root for `python:3-slim`) with no opt-in to a
            less-privileged identity.
      Action: Add a `run_as_user` config field defaulting to a non-root UID
              (e.g. `"1000:1000"`) and pass it to
              `client.containers.create(..., user=run_as_user)`; require an
              explicit human-approval flag (e.g. `allow_root=True`) to
              override the default.

  PRAX-2026-05-26-004  `create_default_code_executor` silently falls back from
            Docker to the local host executor when Docker is unavailable,
            bypassing the remit's human-approval requirement for local
            execution.
      Action: Make the fallback opt-in: replace the silent fall-through with
              `raise RuntimeError("Docker unavailable; pass
              allow_local_fallback=True to use the host executor")` unless the
              caller has explicitly enabled it; preserve the existing behavior
              only behind an explicit kwarg so production callers see a hard
              failure rather than a degraded executor.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-26-005  Local-executor docstring promises a regex-based
            dangerous-command sanitizer that does not exist in the code.
      Action: Either delete the docstring sentence (and replace with an
              explicit statement that no command-level sanitization is
              applied), or implement the promised regex denylist for
              unambiguously destructive shell idioms (e.g. `rm -rf /`,
              `:(){:|:&};:`, `mkfs`, `dd of=/dev/`) and add a test fixture
              that confirms each pattern is rejected.

  PRAX-2026-05-26-006  Docker executor enforces no network policy on the
            spawned container, leaving outbound traffic at the Docker default
            (full bridge egress).
      Action: Add a `network_mode` config field defaulting to `"none"` (no
              network) and require explicit operator opt-in to enable network
              access; document the kwarg as the configured allow-list
              mechanism the remit calls for.

  PRAX-2026-05-26-007  Docker executor accepts caller-supplied `extra_volumes`
            and `extra_hosts` with no validation against the work-directory
            confinement invariant.
      Action: Validate `extra_volumes` against a denylist of sensitive host
              paths (`/var/run/docker.sock`, `/`, `/etc`, `/proc`, `/sys`,
              `/dev`) and reject by default; require an explicit
              `allow_unsafe_volumes=True` constructor kwarg to bypass the
              check, surfaced in the docstring as the remit's human-approval
              signal.

  PRAX-2026-05-26-008  `DockerJupyterServer` sets the host bind directory to
            mode 0o777, exposing executor state files to any local user.
      Action: Replace the 0o777 chmod with the minimal permissions required by
              the container's NB_USER (typically 0o755 for the directory,
              owned by the appropriate UID:GID); if the original intent was to
              work around a docker-stacks UID mismatch, document the actual
              UID and chown to it instead of widening the permission bits.

  PRAX-2026-05-26-009  `JupyterCodeExecutor` (non-Docker variant) runs
            LLM-generated code in the caller's own Python process with no
            isolation boundary.
      Action: Rename or deprecate `JupyterCodeExecutor` in favor of
              `DockerJupyterCodeExecutor` for any production use, and have the
              class emit a `UserWarning` on `__init__` mirroring the one on
              `LocalCommandLineCodeExecutor` so callers see the trust boundary
              at instantiation.

  PRAX-2026-05-26-010  `PowerShell` invocation in the local executor uses
            `-ExecutionPolicy Bypass`, defeating the host's PowerShell
            execution policy.
      Action: Replace `-ExecutionPolicy Bypass` with `-ExecutionPolicy
              RemoteSigned` (or `AllSigned` for high-trust deployments) and
              make the level a configurable kwarg with `Bypass` requiring an
              explicit `allow_unsigned_powershell=True` opt-in.
