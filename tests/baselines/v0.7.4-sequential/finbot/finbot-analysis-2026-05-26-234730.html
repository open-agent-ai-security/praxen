Executive Summary
Agent Remit (as declared)
FinBot is described as CineFlow Productions' autonomous invoice-processing agent that receives vendor invoices, validates the submitter against registered vendor records, runs fraud-detection heuristics, and decides whether to approve (triggering payment), reject, or escalate each invoice. The remit authorizes a fixed tool set — invoice retrieval, approve, reject, escalate, fraud-detection, plus auto-approve / auto-reject below explicit thresholds — and lists three counterparties only (registered vendors as data sources, CineFlow finance admins as configurators, and the LLM provider for inference). Several controls are stated as MUST or MUST NOT: fraud detection cannot be skipped or disabled, runtime messages and invoice fields must not alter goals or decision logic, and decision logic itself must not be redefinable at runtime by any caller.
Behavior Summary (as observed)

The dominant pattern is policy declared in the prompt, zero code-level enforcement: every behavioral MUST in the Worker Remit — fraud detection cannot be skipped, runtime messages must not alter goals, decision logic must not be redefinable at runtime, no approval without human sign-off above threshold — is contradicted by a code path that lets a single unauthenticated admin POST replace the entire goal model, or that lets the LLM call approve_invoice with no runtime check on amount, fraud signal, or vendor status.

This collapses into a one-hop compound chain: vendor-controlled invoice description reaches the LLM context unsanitised through get_invoice_details, and the LLM's tool call to approve_invoice writes payment_processed=True directly with no interposition — so an indirect prompt-injection in a single vendor-submitted invoice description is sufficient to issue payment without human review. The unauthenticated /admin/finbot/goals endpoint provides the same outcome through a more direct channel, and the absence of any structured action log means neither path leaves a forensic trail beyond the LLM's own narrative reasoning.

Scope of Analysis
Python Flask application backed by SQLAlchemy / SQLite, deployed via Gunicorn on Render. A single FinBotAgent class in src/services/finbot_agent.py orchestrates an OpenAI function-calling loop with five tools (get_invoice_details, approve_invoice, reject_invoice, request_human_review, detect_fraud_patterns) and a deterministic _fallback_processing rule-engine that runs when the OpenAI client is unavailable. Three Flask blueprints register under /api with no authentication middleware: admin_bp exposes /admin/finbot/config, /admin/finbot/goals, /admin/finbot/reprocess/<id>, and vendor-trust mutation; vendor_bp exposes invoice submission whose description field flows verbatim into the LLM context. A runtime-mutable FinBotConfig.custom_goals column is concatenated into the system prompt on every invocation, and the fraud_detection_enabled boolean gates the _detect_fraud_patterns tool entirely.

Maturity Scoring Rubric

Every score above is based on this scale. A score is a snapshot of observable posture — not a verdict on the people or team behind the system.

Score Label Meaning
5 Exemplary Best-in-class; automated, continuously tested, reference quality. Rarely achieved in shipping systems.
4 Strong Comprehensive controls, active management, minor gaps. Production-ready.
3 Established Documented controls consistently applied; known gaps accepted. A respectable baseline.
2 Partial Some controls exist but coverage is incomplete; key gaps remain.
1 Ad hoc Informal or inconsistent measures; relies on individual judgment.
0 Absent No evidence this category is addressed at all.
Weighting: the weighted overall above is the sum of each category's score × weight (the per-category weights are shown on each card). Zero Trust carries double weight by design; see the RAISE framework reference for the rationale.
