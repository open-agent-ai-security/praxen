BEHAVIOR SUMMARY
------------------------------------------------------------------------------
Devika's dominant pattern is policy declared in prose, zero code-level
enforcement — and the gap concentrates on the most consequential remit
clauses. The remit promises a sandboxed runner; src/sandbox/code_runner.py and
src/sandbox/firejail.py are both zero-byte stubs while Runner.run_code calls
subprocess.run directly on LLM-produced command strings with no allowlist,
approval gate, or step cap. The remit forbids file I/O outside the project
work directory; Coder.save_code_to_project writes
os.path.join(self.project_dir, project_name, file['file']) with no
path-traversal check, and the Flask /api/get-browser-snapshot endpoint passes
a user-supplied snapshot_path straight into send_file. The remit forbids
communication beyond LLM and search providers; the Decision agent advertises
Twitter/Reddit posting as a legitimate browser_interaction use case, the
Action handler invokes Netlify().deploy(), and the API server binds to 0.0.0.0
on every interface with no authentication, exposing settings rewrite, project
deletion, and arbitrary-path file read to anyone on the network. The compound
chain is straightforward and present end-to-end in the source: a user prompt
or attacker-controlled web page (no injection screen) reaches the Coder/Runner
agents, the LLM emits a commands[] array, and subprocess.run executes those
commands as the application user with no audit trail beyond a free-form
devika_agent.log. There are no tests, no red-team artifacts, and ROADMAP.md
lists "Create an extensive testing suite" as still-unchecked.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-26-001  LLM-produced commands are executed via `subprocess.run`
            with no sandbox, allowlist, approval gate, or step cap.
      Action 1: In `src/agents/runner/runner.py:run_code`, replace the direct
                `subprocess.run` invocation with a call into an
                actually-implemented sandbox runner (firejail/nsjail/container
                exec) that takes a per-command allowlist; for any command
                outside the allowlist, route through a synchronous
                operator-approval gate before execution and record the command
                + outcome to the per-project audit log.
      Action 2: Reject commands whose first token is not in a small allowlist
                (`python3`, `node`, `npm`, `go`, `cargo`) until an explicit
                operator-approved expansion is recorded; treat `pip install`,
                `apt`, `sudo`, `curl | sh`, and any redirection as forbidden
                by default and require per-invocation approval.

  PRAX-2026-05-26-002  The sandbox modules named as the safety boundary are
            zero-byte stubs — the declared isolation primitive does not exist
            as code.
      Action 1: Implement `src/sandbox/firejail.py` (or replace with an
                `nsjail`/container-exec wrapper) before any further use of
                `Runner.run_code` in a deployed configuration; wire the Runner
                to call into the sandbox wrapper rather than `subprocess.run`
                directly.
      Action 2: Until the sandbox is implemented, gate the `/api/run-code`
                route and the `action == "run"` branch in `subsequent_execute`
                behind a feature flag that defaults to off, and surface the
                unimplemented-sandbox state to the operator in the UI.

  PRAX-2026-05-26-003  Flask + Socket.IO server binds to `0.0.0.0:1337` with
            zero authentication on every API route and Socket.IO event.
      Action 1: Change the bind host in `devika.py:209` from `0.0.0.0` to
                `127.0.0.1` and document the local-only contract; if a remote
                UI scenario is supported, require an explicit configuration
                flag and bind only to that interface.
      Action 2: Add a Flask `before_request` handler that rejects any request
                without a session token bound to the local operator (e.g., a
                token issued at process start and read from a user-readable
                config file); apply the same check to Socket.IO event handlers
                via `socketio.on_event` wrappers.

  PRAX-2026-05-26-004  `/api/get-browser-snapshot` returns any file on disk
            via `send_file(request.args.get("snapshot_path"))`.
      Action 1: In `devika.py:browser_snapshot`, resolve `snapshot_path`
                against `Config().get_screenshots_dir()` using
                `os.path.realpath` and reject any resolved path whose
                `os.path.commonpath` with the screenshots directory is not the
                screenshots directory; return a 400 in that case.
      Action 2: Replace the `snapshot_path` query parameter with a
                `snapshot_id` that looks up a server-managed mapping table;
                never accept a filesystem path from the client.

  PRAX-2026-05-26-005  Untrusted web content reaches the Coder/Runner agents
            without an injection screen and the resulting LLM output is
            executed as shell commands — full external-input-to-RCE chain.
      Action 1: Insert an injection-screen layer between
                `Browser.extract_text` (or `Formatter.execute`) and any
                consumer that writes the content into an LLM prompt; reject or
                quarantine content containing imperative tokens matching a
                known injection corpus (`ignore previous`, `system:`,
                role-prefix tokens, fenced instruction blocks targeting the
                agent), and surface the rejection to the operator.
      Action 2: Decouple the model's `commands[]` output from execution: route
                every command through the operator-approval gate described in
                PRAX-2026-05-26-001 before `subprocess.run`, so an injection
                that reaches the Coder cannot end in RCE without a
                human-in-the-loop decision.

  PRAX-2026-05-26-006  `Coder.save_code_to_project` writes LLM-emitted
            `file['file']` paths via raw `os.path.join` with no traversal
            check.
      Action 1: In `Coder.save_code_to_project`, compute
                `os.path.realpath(file_path)` and verify it starts with
                `os.path.realpath(os.path.join(self.project_dir,
                project_name)) + os.sep`; refuse the write and log a Critical
                audit event otherwise.
      Action 2: Strip absolute-path prefixes and `..` segments from
                `file['file']` before join; record any stripped input as a
                finding in the project-level audit log so an injection
                attempting filesystem escape is visible to the operator.

  PRAX-2026-05-26-007  Netlify production deployment is wired as a first-class
            action — a counterparty and capability the remit explicitly
            forbids.
      Action 1: Remove the `deploy` branch from `Agent.subsequent_execute` and
                delete `src/services/netlify.py` from the runtime path, OR (if
                deploy is intended scope) update the Worker Remit to authorize
                Netlify as a counterparty and require explicit per-invocation
                operator approval before `Netlify().deploy` is called.
      Action 2: Until the remit is updated, guard the `deploy` branch behind a
                feature flag defaulting to off and route any deploy attempt
                through a synchronous approval prompt to the operator.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-26-008  Decision agent prompt advertises Twitter/Reddit posting
            as a `browser_interaction` use case — domain expansion at the
            prompt layer.
      Action 1: Edit `src/agents/decision/prompt.jinja2` to remove the
                Twitter/Reddit/posting language and restrict
                `browser_interaction` to research-style read-only
                interactions; mirror that constraint in the
                `start_interaction` code path by maintaining an allowlist of
                permitted domains and refusing TYPESUBMIT actions against any
                unlisted host.
      Action 2: Document the authorized search/research domains in the Worker
                Remit's Known Good Baseline and have `start_interaction`
                reject navigation to any host outside that list.

  PRAX-2026-05-26-009  The global `Knowledge` SQLite table has no project
            column — search results from one project are returned to every
            other project's Coder prompt.
      Action 1: Add a non-null `project` column to `Knowledge` and update both
                `add_knowledge` and `get_knowledge` to require `project` as a
                parameter and filter on `(tag, project)`; refuse calls with a
                missing or empty project value.
      Action 2: Either re-enable the now-commented add/get call sites with the
                new project-scoped signatures, or remove the `KnowledgeBase`
                class entirely if the project has decided not to ship it.

  PRAX-2026-05-26-010  No wall-clock or step-count cap on the agent execution
            loop — `retry_wrapper`'s 5-attempt cap is the only bound.
      Action 1: Add a `RunBudget` object to `Agent.execute` and
                `Agent.subsequent_execute` that tracks aggregate steps and
                wall-clock seconds; pass it down into Planner / Researcher /
                Coder / Runner / Action and halt with an operator-visible
                message when either cap is hit.
      Action 2: In `Runner.run_code`, refuse to execute a `commands[]` array
                longer than a configurable `RUNNER_MAX_COMMANDS_PER_RUN`
                (default 10), and surface the truncation to the operator.

  PRAX-2026-05-26-011  Runner prompt invites `pip3 install -r
            requirements.txt` and the Runner executes it with no allowlist or
            operator approval.
      Action 1: Rewrite the runner prompt example to omit `pip install` (or
                replace it with a placeholder that triggers an explicit
                operator-approval flow) and add a deny list in
                `Runner.run_code` that refuses any command beginning with
                `pip`, `pip3`, `apt`, `apt-get`, `brew`, `npm install`, or
                `curl ... | sh` until the operator approves it per-invocation.
      Action 2: Document the allowed-package mechanism in the Worker Remit's
                pre-approved list; require that list to live in code (e.g.,
                `APPROVED_PACKAGES`) and consult it on every install attempt.

  PRAX-2026-05-26-012  No per-project action-level audit log — only a single
            global plaintext application log captures HTTP route entries.
      Action 1: Add a `src/audit.py` writing JSON-lines per-project audit
                files at `data/projects//audit.jsonl` with fields `{ts, actor,
                action_type, target, params, outcome, finding_id}`; instrument
                every subprocess invocation, file write, web fetch, and LLM
                call to emit one record.
      Action 2: Document the audit log location in the Worker Remit's Known
                Good Baseline and add an integrity-check (append-only mode,
                daily checksum) so the audit log is itself protected from
                agent tampering — closing R-04's "MUST NOT modify its own
                memory store" path that the audit file would otherwise share.

  PRAX-2026-05-26-013  All 34 runtime dependencies in `requirements.txt` are
            unpinned — no `==`, no lockfile committed.
      Action 1: Pin every entry in `requirements.txt` to an exact version
                (`==`) and commit a `uv.lock` / `requirements.lock` capturing
                transitive resolutions; rebuild the Docker image against the
                pinned set.
      Action 2: Add a CI job that fails on any unpinned entry and on any drift
                between `requirements.txt` and the committed lockfile; run
                dependency-vulnerability scanning (e.g., `pip-audit`) on every
                PR.

  PRAX-2026-05-26-014  Zero tests, zero red-team artifacts, and `ROADMAP.md`
            lists testing as unchecked — production runtime with no
            adversarial testing of any kind.
      Action 1: Stand up a minimal test suite under `tests/` covering at
                minimum: the path-traversal contract on
                `Coder.save_code_to_project` (PRAX-2026-05-26-006), the
                command-allowlist contract on `Runner.run_code`
                (PRAX-2026-05-26-001), and the project-scope contract on
                `KnowledgeBase` (PRAX-2026-05-26-009). Wire into CI.
      Action 2: Schedule an adversarial exercise that targets the
                search→coder→runner chain (PRAX-2026-05-26-005) with
                prompt-injection payloads in a controlled local environment;
                record findings as architectural change requests, not just
                config tweaks.

  PRAX-2026-05-26-015  `POST /api/settings` lets any unauthenticated caller
            rewrite API keys and runtime config — a credential modification
            primitive over the network.
      Action 1: Add an authentication check in front of `/api/settings` (POST
                and GET) using the same local-operator token introduced for
                PRAX-2026-05-26-003; reject anonymous calls.
      Action 2: In `Config.update_config`, refuse to write `API_KEYS.*` and
                `API_ENDPOINTS.*` over HTTP — restrict those to the on-disk
                `config.toml` only, edited by the operator out-of-band.

  PRAX-2026-05-26-016  `start_interaction` browser loop feeds raw page text to
            the LLM and executes the LLM's CLICK/TYPE output against the live
            page — indirect prompt injection → in-browser action chain.
      Action 1: Refuse `start_interaction` navigation to any host not in a
                Worker-Remit-authorized list of research domains; refuse
                `TYPESUBMIT` actions on any host outside that list.
      Action 2: Insert an injection-screen on `browser_content` before it is
                interpolated into `prompt_template`, and require operator
                approval for the first `CLICK`/`TYPE` action on a host the
                agent has not interacted with before in this project.
