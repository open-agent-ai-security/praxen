BEHAVIOR SUMMARY
------------------------------------------------------------------------------
CraftBot's safety model is almost entirely declarative: the Permission Tiers
and approval gates in `PROACTIVE.md` and `AGENT.md`, the injection-defense
paragraph in the system prompt, and the unwired `PromptSanitizer` all describe
controls the executor never enforces. No code path can block a `send`, a
payment, a `task_end`, or a shell command pending human approval — the
`irreversible` flag drives only crash-recovery idempotency — and external web,
inbound-message, and memory content flows into the LLM context with no trust
boundary. That soft posture sits on top of an unauthenticated WebSocket
control plane (`/ws` on `localhost:7926`, no auth or Origin check) whose
messages include file CRUD, settings changes, and `mcp_add_json`
arbitrary-server registration — reachable by any local process or, via
cross-site WebSocket hijacking, any website the operator visits. The net
result is a short, real exploit chain: injected web or message content, or a
hostile local page, reaches the agent's context or control plane and drives an
auto-executed shell command, `send`, or payment with no isolation and the
operator's full host environment. Start with the approval gate, execution
isolation, and `/ws` authentication.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-07-10-001  The `/ws` WebSocket on localhost:7926 is an
            unauthenticated agent control plane reachable by any local process
            or, via cross-site WebSocket hijacking, any website.
      Action 1: Require a per-instance auth token on the `/ws` handshake and
                reject connections whose `Origin` header is not the local UI,
                closing the cross-site WebSocket hijacking path in
                `_websocket_handler`.
      Action 2: Gate state-changing WebSocket messages (`settings_update`,
                `mcp_add_json`, file writes) behind that authenticated session
                rather than accepting them from any connected client.

  PRAX-2026-07-10-002  Untrusted external content flows into the LLM context
            with no validation and can drive an auto-executed shell command,
            send, or payment in one hop.
      Action 1: Add a deterministic trust boundary on ingested external
                content (label tool/web/message output as untrusted and strip
                or quarantine instruction-like spans) rather than relying on
                the `context.py:142` prompt paragraph.
      Action 2: Interpose a code-level approval gate before shell/send/payment
                execution so injected content cannot reach a high-impact
                action without operator confirmation.

  PRAX-2026-07-10-003  No code-level approval gate exists for irreversible or
            high-impact actions; the Permission Tiers and complex-task
            approval are prompt guidance only.
      Action 1: Implement a real approval gate in the executor keyed on action
                metadata (irreversible/high-impact) that blocks execution
                until an operator confirmation is recorded, and make
                `permission_tier` 2-3 actually route through it.
      Action 2: Add a code check that a complex task's approval was recorded
                before `end_task` may complete it.

  PRAX-2026-07-10-004  Arbitrary shell and code execution runs automatically
            with no approval and no host isolation, inheriting the operator's
            full secret-bearing environment.
      Action 1: Run shell/code execution inside real isolation (container or
                sandbox with restricted filesystem, network, and a minimal
                environment) instead of a host venv, and stop passing
                `os.environ.copy()` into the subprocess.
      Action 2: Require operator approval for state-changing or destructive
                commands, or apply a command policy, before `run_shell`
                executes.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-07-10-005  The filesystem MCP server is enabled by default with
            root scope "." — full read/write over the project including
            `.credentials/` and `settings.json` — and its tools have no
            approval gate.
      Action 1: Scope the filesystem MCP server to a narrow working directory
                (not `.`) that excludes `.credentials/` and `app/config/`, or
                disable it by default.
      Action 2: Route high-risk MCP tools (write/delete/exec/send) through the
                same approval gate recommended for built-in high-impact
                actions.

  PRAX-2026-07-10-006  Inbound integration messages reach the agent's react
            loop with no authorized-counterparty check; the only protection is
            a soft prompt wrapper.
      Action 1: Enforce an authorized-counterparty allowlist per platform in
                code, dropping or queuing-for-approval messages from senders
                not on it, rather than relying on the prompt wrapper.
      Action 2: Default `auto_reply` to off until an identity check is in
                place.

  PRAX-2026-07-10-007  No enforced input validation on external content; the
            purpose-built `PromptSanitizer` is dead code and the only active
            defense is one system-prompt paragraph.
      Action 1: Wire a deterministic sanitization/quarantine step into the
                ingestion path (web_fetch, web_search, inbound-message
                handling) that trust-labels or strips instruction-like content
                before it enters context.
      Action 2: Either make `PromptSanitizer` block (not just log) and call
                it, or remove it so it does not read as an implemented
                control.

  PRAX-2026-07-10-008  Session-loaded identity and memory files are
            agent-writable with no validation or approval, giving injected
            content a one-hop persistence path (ASI06).
      Action 1: Require operator confirmation (a code gate) before any write
                to `SOUL.md`/`USER.md`/`AGENT.md`/`PROACTIVE.md` or the memory
                store.
      Action 2: Validate/quarantine distilled memory content before it is
                persisted and re-injected, so poisoned event text cannot
                become a standing instruction.

  PRAX-2026-07-10-009  No evidence of adversarial or security testing anywhere
            in the project; the test suite is functional only and CI is
            disabled.
      Action 1: Add an adversarial test suite covering indirect prompt
                injection via web and inbound-message channels and shell/exec
                abuse, and run it in CI.
      Action 2: Re-enable CI with at least a SAST gate, and feed findings back
                into architectural fixes (approval gate, isolation, input
                validation).
