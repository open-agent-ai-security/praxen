BEHAVIOR SUMMARY
------------------------------------------------------------------------------
Aider's safety model is "developer-in-the-loop": almost every consequential
action — shell execution, commit acceptance after `/run`, applying an LLM edit
to a file not in the chat — passes through `IO.confirm_ask()` before it lands.
That control is real and runs on the agent's primary edit path, which lifts it
out of the Absent band. The dominant pattern, however, is that the *gates
around* the gates are weak — --yes-always globally collapses every confirm to
"yes", --git-commit-verify defaults to off so commits ship with --no-verify by
default, the auto_commit hook fires after every applied edit with no
diff-accept prompt of its own, and abs_root_path() never re-checks that a
resolved path stays inside the repo root. The single most load-bearing
structural gap is the absence of any secret scanner anywhere in base_coder.py,
io.py, repo.py, or commands.py, despite the remit's explicit MUST that
secret-like strings be redacted before they reach LLM context, proposed edits,
or commit messages.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-001  `--watch-files` mode auto-triggers an LLM turn from `#
            ai!` comments in the source tree, treating untrusted file content
            as operator instructions with no injection-detection step.
      Action 1: In `watch.py:process_changes`, gate the LLM dispatch on an
                explicit developer keypress in the terminal session (not just
                the file save) before composing the `watch_code_prompt`.
      Action 2: Neutralize `# ai!` markers that appear in files newly added to
                chat or recently pulled — only honor markers that the
                developer authored in the *current* editing session.

  PRAX-2026-05-29-002  `abs_root_path()` resolves paths without verifying
            containment in the repo root, and `/add` / `/read-only` accept
            absolute and tilde-expanded paths, contradicting the MUST that
            file I/O stay inside the repo.
      Action 1: Add a
                `Path(resolved).is_relative_to(Path(self.root).resolve())`
                check inside `abs_root_path()` and refuse (raise / return
                None) on failure for every caller.
      Action 2: In `cmd_add` and `cmd_read_only`, reject absolute and
                tilde-expanded paths unconditionally (no developer prompt —
                the remit says "refused, not merely confirmed") and treat
                symlinks via `Path.resolve(strict=False)` before the
                containment test.

  PRAX-2026-05-29-003  `auto_commit` writes an aider commit after every
            applied LLM edit with no diff-accept prompt, defeating R-19's
            "developer must see the diff and accept it before the commit is
            made".
      Action 1: In `Coder.auto_commit`, gate the `self.repo.commit(...)` call
                on `io.confirm_ask('Commit these changes?', subject=,
                default='y', explicit_yes_required=True)` and surface the diff
                in the subject, with a `--no-confirm-commit` opt-out the
                developer must set deliberately.
      Action 2: Document the new default in the README's safety section and in
                `--help` for `--auto-commits`.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-004  `--yes-always` globally collapses every
            `io.confirm_ask` to "yes", turning the entire confirm-prompt
            safety model into a no-op with a single flag.
      Action 1: Split `--yes-always` into per-category flags
                (`--auto-confirm-add`, `--auto-confirm-edit`) and refuse to
                auto-confirm shell execution or commits regardless of flag
                state.
      Action 2: On startup with `--yes-always`, print a red banner naming the
                specific gates that are now disabled and require a
                re-confirmation on first session of each repo.

  PRAX-2026-05-29-005  `--git-commit-verify` defaults to False, so every aider
            commit ships `git commit --no-verify` and bypasses pre-commit
            hooks unless the developer opts in.
      Action 1: Flip the `--git-commit-verify` default to `True` so aider
                commits honour project pre-commit hooks by default; require an
                explicit `--no-git-commit-verify` to skip them.
      Action 2: On the first commit per session, surface a one-line message
                naming whether hooks ran and what they returned.

  PRAX-2026-05-29-006  `analytics.py` ships hardcoded Mixpanel and PostHog
            client tokens that send event telemetry off the developer's
            machine to vendor endpoints — a destination the remit's R-05 does
            not authorize.
      Action 1: Either flip the analytics default to off (no transmission
                until the developer explicitly opts in) or update the remit
                (R-05) to list Mixpanel and PostHog as authorized telemetry
                destinations.
      Action 2: Add a startup banner naming the telemetry destinations
                actively in use, so the developer learns about R-05's third
                destination without reading the source.

  PRAX-2026-05-29-007  `/read-only` and `/add` accept absolute paths to
            credential-adjacent files (`.env`, `~/.ssh/*`,
            `~/.aws/credentials`) with no special handling, violating R-13's
            MUST NEVER on credential-bearing files.
      Action 1: Add a credential-pattern denylist to `cmd_add` and
                `cmd_read_only` (matching `.env*`, `*.pem`, `*id_rsa*`,
                `*credentials*`, `*secrets*`, `*token*`) that refuses the
                addition unless the developer types a phrase like "I
                understand this file may contain secrets" — i.e. an explicit
                acknowledgment as the remit requires.
      Action 2: Log every credential-pattern path the developer overrides to
                the chat history with a redaction marker.

  PRAX-2026-05-29-008  No secret scanner exists anywhere in scope, leaving the
            remit's MUST to redact secret-like strings from LLM context,
            proposed edits, and commit messages entirely unenforced.
      Action 1: Add a pre-commit redaction pass in `Repo.commit` that runs the
                diff through a secret-pattern detector (`gitleaks`-style
                rules) and refuses the commit if a hit is found, with an
                explicit `--allow-secret` override.
      Action 2: Add the same pre-add redaction pass in `cmd_add` /
                `cmd_read_only` / `cmd_paste` / `cmd_web` so secret-bearing
                content never enters LLM context in the first place.
