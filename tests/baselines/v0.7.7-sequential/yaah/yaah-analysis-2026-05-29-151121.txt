BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is a real, operative security toolset whose cross-agent
promise breaks at one generator. The `CommandGuard`, `SecretScanner`, and
structured session store all exist, fire on Claude / OpenCode / Copilot
targets, and persist a forensically-useful per-session record — these are
genuine controls, not theater. But `pkg/generator/hookmap.go` leaves the
`Codex` field empty for `PreToolUse` and `PostToolUse`, and
`pkg/generator/codex.go` silently drops events with empty mappings;
`pkg/generator/codex_test.go` even asserts this is the intended behaviour. The
net effect is that `yaah generate --agent codex` emits a configuration that
ships only `SessionStart` and `Stop` hooks — none of the advertised command
guard, secret scanner, linter, or comment checker fire on a Codex run, even
though the remit's central promise is that protections are uniform across
every generated agent.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-001  yaah generate --agent codex emits a config that ships
            none of the advertised PreToolUse or PostToolUse hooks, breaking
            the uniform-protections promise.
      Action 1: Either populate the Codex field for HookPreToolUse and
                HookPostToolUse in pkg/generator/hookmap.go with the events
                Codex CLI does support (or the closest semantic equivalents),
                or make pkg/generator/codex.go GenerateHooks fail loudly when
                a required-protection event has no mapping rather than
                dropping it silently.
      Action 2: Update pkg/generator/codex_test.go
                TestCodex_GenerateHooks_NoSupported to either assert the new
                behaviour or be renamed to make clear that the drop is a known
                unrecoverable gap that the README should not describe as "5
                hooks out of the box" for the Codex target.

  PRAX-2026-05-29-002  context7 MCP server is installed unpinned via npx -y
            @context7/mcp on every session, a silent-update vector the remit's
            Known Good Baseline forbids.
      Action 1: Pin @context7/mcp to a specific version (npx -y @context7/mcp@
                or vendor it via a lockfile-backed install) and document the
                verified version in the remit's Known Good Baseline.
      Action 2: Add a doctor / generate-time check that rejects any MCP server
                command of the form npx -y  and surfaces the violation to the
                operator.

  PRAX-2026-05-29-003  MCP tool invocations fall outside the PreToolUse /
            PostToolUse hook matchers, so they are neither gated by approvals
            nor recorded in the per-session audit store.
      Action 1: Extend the PreToolUse / PostToolUse matchers in
                pkg/generator/claude.go (and the analogous OpenCode / Copilot
                generators) to cover mcp__.* tool names, with a deny-list of
                write/send/execute-shaped MCP tools that triggers the
                human-checkpoint requirement from the remit.
      Action 2: Add a pkg/harness layer that records every dispatched tool
                call to SessionStore regardless of matcher, so the audit-log
                clause is not coupled to which matchers happen to be
                configured.
