BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is sandbox-shaped-not-sandbox-strong: every executor
preserves the *shape* of isolation (a container exists, a temp work-dir
exists, a timeout is configured) but defaults strip away the *substance* â€”
Docker containers run as the image's default user (typically root in
python:3-slim) with no read_only rootfs, no mem_limit, no cap_drop, and full
network access; the local executor copies os.environ wholesale into the
subprocess so any host credential the parent process holds becomes available
to LLM-generated code; create_default_code_executor() answers a
DockerException on client.ping() with a warnings.warn and a return of
LocalCommandLineCodeExecutor, turning a hardened default into the unhardened
one with no operator confirmation; and JupyterCodeExecutor wraps the cell in
asyncio.wait_for(asyncio.shield(execute_task), timeout=self._timeout) which
raises TimeoutError in the caller but leaves the kernel cell running.
Compounding the posture, the remit mandates a per-execution audit log
(timestamp, executor kind, source agent, work-dir, timeout, exit status, code
digest) and none of the five executors emits one â€” only setup-time
logging.info calls for package install and image pull exist. The combination
of "host env into subprocess" + "no audit log" + "silent Dockerâ†’Local
downgrade" is the chain a security lead should look at first: LLM-generated
code can read host credentials and the operator has no record of which
executor ran or what it did.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-001  `create_default_code_executor()` silently downgrades
            from the containerised executor to the local-host executor on any
            Docker failure with only a `UserWarning`.
      Action 1: In `code_executors/__init__.py`, replace the silent fallback
                with an explicit opt-in: require an env flag such as
                `AUTOGEN_ALLOW_LOCAL_EXECUTOR=1` (or an explicit
                `allow_local_fallback=True` argument) before returning
                `LocalCommandLineCodeExecutor`; otherwise raise `RuntimeError`
                naming the missing Docker prerequisite.
      Action 2: Surface the chosen executor kind in the function's return
                value or via a structured log line so the calling agent and
                the operator can both verify which backend ran.

  PRAX-2026-05-29-002  `LocalCommandLineCodeExecutor` copies the parent
            process's full `os.environ` into the executed subprocess, exposing
            every host credential the parent holds to LLM-generated code.
      Action 1: Replace `os.environ.copy()` with an explicit allow-list (e.g.
                `PATH`, `LANG`, `HOME`, plus user-supplied variables) and
                document the variables that survive the boundary; reject calls
                that pass through known credential-shaped names (`*_KEY`,
                `*_TOKEN`, `*_SECRET`, `AWS_*`, `AZURE_*`, `GOOGLE_*`) without
                explicit opt-in.
      Action 2: Make the env-passing surface a constructor argument (`env:
                Optional[Dict[str, str]]`) so the operator must consciously
                hand variables across the trust boundary.

  PRAX-2026-05-29-003  None of the five executor implementations emits a
            per-execution audit record despite the remit naming eight fields
            that MUST be captured per call.
      Action 1: Add a structured audit emitter to `_common.py` â€” a
                `log_execution(executor_kind, language, source_agent,
                work_dir, timeout, exit_code, code_digest, started_at,
                finished_at)` helper that writes one JSON line per execution,
                configurable via env or constructor argument, defaulting to
                enabled.
      Action 2: Wire the helper into every `_execute_code_dont_check_setup`
                (Local, Docker, Jupyter, DockerJupyter, Azure) and into the
                cancellation paths so timeouts and cancellations are recorded
                too.

  PRAX-2026-05-29-004  Docker containers are created with none of the standard
            hardening flags (`user=`, `read_only=`, `mem_limit=`, `cap_drop=`,
            `network_mode=`), so LLM-generated code runs as the image's
            default user (usually root in `python:3-slim`) on a writable
            rootfs with full Linux capabilities and unrestricted network
            egress.
      Action 1: Set hardening defaults on
                `containers.create`/`containers.run`: `user="nobody"` (or an
                unprivileged numeric uid in the image), `read_only=True` with
                a tmpfs over `/tmp`, a default `mem_limit` (e.g. 512m) and
                `pids_limit`, `cap_drop=["ALL"]`,
                `security_opt=["no-new-privileges:true"]`, and
                `network_mode="none"` for the command executor; expose
                constructor arguments to override each, with each override
                requiring an explicit opt-in flag.
      Action 2: Replace `publish_all_ports=True` in `DockerJupyterServer` with
                an explicit single-port publish bound to `127.0.0.1` so the
                Jupyter port is not advertised on all host interfaces.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-005  Work-directory confinement is only enforced for `#
            filename:` comment directives; the executed code itself can read
            or write any path the executor process can reach, and command-line
            executors `cd` into the work_dir without applying a chroot, mount
            namespace, or seccomp filter that would bind I/O to it.
      Action 1: Document that `LocalCommandLineCodeExecutor` does not confine
                I/O to the work directory and gate its construction behind the
                same approval flag as R-13 (see PRAX-2026-05-29-001);
                recommend Docker/Jupyter for any deployment where confinement
                is required.
      Action 2: In `DockerCommandLineCodeExecutor`, set `read_only=True` on
                the container (paired with PRAX-2026-05-29-004) and mount the
                work_dir as the only writable path, so absolute writes outside
                `/workspace` fail at the kernel layer instead of silently
                succeeding on the rootfs.

  PRAX-2026-05-29-006  `JupyterCodeExecutor` wraps cell execution in
            `asyncio.wait_for(asyncio.shield(execute_task),
            timeout=self._timeout)`, which raises `TimeoutError` to the caller
            but leaves the kernel cell running on the Jupyter server â€” the
            timeout is observed by the orchestrator but not enforced at the
            execution boundary.
      Action 1: On timeout in `JupyterCodeExecutor._execute_code_block`, send
                a kernel interrupt (e.g. `kc.interrupt()` or `await
                self._client.km.interrupt_kernel()`) and, if interrupt does
                not complete within a short grace window, restart the kernel â€”
                preserving the remit's contract that processes exceeding the
                timeout are terminated.
      Action 2: Remove the `asyncio.shield` wrapper or replace it with a
                paired `execute_task.cancel()` + `await
                self._await_with_grace(execute_task)` so the in-process task
                is also cancelled when the caller observes a timeout.

  PRAX-2026-05-29-007  No executor restricts outbound network access from the
            execution environment, and the Docker executor's container is
            created with no `network_mode=` argument so it joins the default
            bridge with full egress; the remit requires that any egress beyond
            an allow-list have human approval.
      Action 1: Default `network_mode="none"` for
                `DockerCommandLineCodeExecutor` and expose `network_mode` (or
                an `allow_network: bool` plus `network_allowlist: list[str]`)
                as an explicit constructor argument that must be opted into;
                document that enabling egress requires the operator's R-14
                approval.
      Action 2: Move `extra_hosts` behind the same allow-network opt-in, and
                remove `publish_all_ports=True` in `DockerJupyterServer` in
                favour of an explicit single-port publish bound to
                `127.0.0.1`.

  PRAX-2026-05-29-008  `DockerCommandLineCodeExecutor` accepts an
            `extra_volumes` constructor argument that is merged unchanged into
            the container `volumes=` mapping, allowing arbitrary host paths to
            be mounted into the container with no approval gate â€” the remit
            requires human approval for any mount beyond the configured work
            directory.
      Action 1: Require an `allow_extra_volumes: bool` (default `False`)
                constructor flag before `extra_volumes` takes effect; raise
                `ValueError` if the caller supplies `extra_volumes` without
                setting the flag, and emit a structured log line listing every
                host path being mounted at start time so the operator can
                audit it.
      Action 2: Reject any `extra_volumes` host path that resolves under a
                denylist (`/etc`, `/root`, the caller's home directory,
                `/var/run/docker.sock`); document the denylist as the minimum
                guardrail and refer the operator to R-15 for case-by-case
                approval.
