BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in the system prompt, zero code-level
enforcement. The Worker Remit's central MUST NOTs — no DML, no DDL, no admin
commands, no multi-statement queries, no out-of-scope tables — are reduced in
code to a single English-language sentence in SQL_PREFIX ("DO NOT make any DML
statements") that the LLM may or may not honor. QuerySQLDatabaseTool._run
calls db.run_no_throw(query) which forwards the model-generated string
straight to connection.execute() inside engine.begin() — no SQL parser, no
statement-type check, no per-table allowlist, no read-only enforcement. The
maintainers know this: a prominent !!! warning admonition at the top of
create_sql_agent tells operators the agent "can execute arbitrary SQL," names
"unintended writes depending on your database permissions" as a risk,
recommends least-privilege roles, and redirects new production work to Deep
Agents. The remit's defense-in-depth clauses are the right shape — but they
are operator responsibilities, not library guarantees. Two operative controls
do exist in the library: a hard max_iterations=15 cap on tool-call loops in
AgentExecutor, and a max_string_length=300 per-cell truncation in
SQLDatabase.run.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-001  DML prohibition is prompt-only; no code-level gate
            prevents data-modifying statements from reaching the database.
      Action 1: Add an opt-in statement-type gate to `QuerySQLDatabaseTool`
                (e.g., a `read_only: bool = False` constructor arg that, when
                true, parses the model-generated SQL with `sqlglot` and
                rejects anything other than SELECT/CTE before calling
                `db.run_no_throw`), and document that the prompt-only
                DML-prohibition is not sufficient on its own.
      Action 2: Cross-reference the existing maintainer warning in
                `create_sql_agent`'s docstring to call out specifically that
                the library does not parse or filter SQL by statement type —
                the operator must enforce read-only at the database role.

  PRAX-2026-05-29-002  DDL and administrative statements have no code-level
            gate — DROP, CREATE, ALTER, GRANT, TRUNCATE all reach the database
            if the LLM emits them.
      Action: Extend the proposed read-only gate in `QuerySQLDatabaseTool` to
              reject any non-SELECT statement family — including DDL,
              GRANT/REVOKE, TRUNCATE, ATTACH/COPY, and dialect-specific
              administrative commands — and surface the rejection back to the
              LLM as a tool error.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-29-003  No multi-statement / stored-procedure guard — concealed
            DML can hide inside a query the LLM treats as a single SELECT.
      Action: Add statement-splitting and count enforcement (single statement
              only) to the same read-only gate proposed in
              PRAX-2026-05-29-001, rejecting multi-statement strings and bare
              `CALL`/`EXEC` invocations.

  PRAX-2026-05-29-004  Query-checker is an LLM self-review, not a
            deterministic gate — and using it is left to the LLM's discretion.
      Action: If the project chooses to keep the LLM-driven checker,
              complement it with a deterministic statement-type gate in
              `QuerySQLDatabaseTool` so the architectural double-check is
              real; otherwise document the checker as advisory only.

  PRAX-2026-05-29-005  Row contents, table names, and column names flow into
            the LLM context without provenance labelling — indirect prompt
            injection via database content.
      Action 1: Document the indirect-injection vector explicitly in
                `create_sql_agent` and the toolkit docs alongside the existing
                maintainer warning, naming `table_info`, `table_names`, and
                row contents as untrusted input from the agent's perspective.
      Action 2: Where the operator controls schema, recommend (and where
                possible enforce) that table-info responses be wrapped with an
                explicit "data, not instructions" delimiter the LLM is trained
                to honor — a partial mitigation, not a fix.

  PRAX-2026-05-29-006  Zero logging on the query-execution path — generated
            SQL, tool calls, and results are not recorded by the library.
      Action: Add a module-level `logger = logging.getLogger(__name__)` to
              `utilities/sql_database.py` and `tools/sql_database/tool.py`,
              emit structured INFO-level events on query execution (statement
              type, table-name set, row count, execution-time bucket) and
              WARNING-level events on errors, so the operator's logging stack
              can capture an audit trail without the library prescribing a
              destination.

  PRAX-2026-05-29-007  No adversarial testing of the SQL agent path —
            `QuerySQLDatabaseTool`, `QuerySQLCheckerTool`,
            `SQLDatabaseToolkit`, and `create_sql_agent` have no unit tests at
            all in the in-scope test directory.
      Action: Add an adversarial unit test that drives `create_sql_agent`
              end-to-end with a stub LLM emitting representative violation
              strings (DML, DDL, multi-statement, table-content prompt
              injection) and asserts the library's response (rejection, error
              message, or operator-visible warning); the test should fail
              today and continue to fail until the library adds a code-level
              gate.
