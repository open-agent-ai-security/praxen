Executive Summary
Agent Remit (as declared)
An autonomous invoice-processing agent for CineFlow Productions' finance team. The remit authorizes FinBot to retrieve invoice and vendor records, run fraud-detection heuristics, and then approve, reject, or escalate each invoice — where approval triggers payment processing. It is constrained to a narrow set of MUST/MUST NOT rules: fraud detection must run before any approval and may never be skipped; the submitting vendor must be registered and approved; any invoice above the manual-review threshold, from a low-trust vendor, carrying a fraud signal, or below the confidence threshold must be escalated to a human; and runtime content — invoice descriptions, vendor notes, payloads — must not alter FinBot's goals or decision logic. FinBot communicates only with its invoice database and its LLM provider, sends no email, and does not create, modify, or delete vendor records.
Behavior Summary (as observed)

FinBot's declared decision boundaries exist almost entirely as English sentences in an LLM prompt, with no deterministic code gate behind any of them. The dominant pattern is total policy-implementation divergence: every MUST / MUST NOT rule in the remit — fraud-detection-always-on, escalate-on-any-fraud-signal, no-runtime-goal-changes, human-sign-off above threshold — is contradicted by the code.

The single most dangerous chain is the compound goal-hijack path: an unauthenticated POST /api/admin/finbot/goals endpoint writes attacker-supplied natural language into config.custom_goals, which get_system_prompt() then splices into FinBot's system prompt under an explicit 'OVERRIDE ABOVE IF CONFLICTING' header — and the same unauthenticated config endpoint can flip fraud_detection_enabled to false, after which both agent paths skip injection detection. Second theme: _approve_invoice sets payment_processed=True with no check on amount, vendor registration status, fraud result, or confidence — the LLM's word is the only thing standing between an invoice and a payment.

Scope of Analysis
A Python Flask application with a SQLAlchemy-backed SQLite database (src/main.py, src/models/vendor.py). A single FinBotAgent class in src/services/finbot_agent.py orchestrates OpenAI function-calling (gpt-4.1-mini) over five tools — get_invoice_details, approve_invoice, reject_invoice, request_human_review, detect_fraud_patterns — with a rule-engine fallback (_fallback_processing) used whenever the OpenAI client is unavailable. The agent's entire goal set is a natural-language system prompt built at runtime in get_system_prompt(), and a custom_goals config field is concatenated into that prompt verbatim under the literal header 'CUSTOM GOALS (OVERRIDE ABOVE IF CONFLICTING)'. Admin routes are exposed under /api/admin/* via a Flask blueprint with no authentication middleware — including POST /api/admin/finbot/goals and POST /api/admin/finbot/config. Vendor-submitted invoice descriptions flow directly into the LLM context through process_invoice(). The repo contains a CTF attack walkthrough, embeds CTF flag strings in finbot_agent.py, has a hardcoded Flask SECRET_KEY in src/main.py, and ships no logging framework and no tests.

CRITICAL PRAX-2026-05-19-001 Unauthenticated POST /api/admin/finbot/goals rewrites FinBot's system prompt at runtime through the custom_goals field.
Implement Zero Trust LLM01 — Prompt Injection ASI01 — Agent Goal Hijack
Policy Rule — R-16, R-17 (Worker Remit):
"Operator configuration MUST NOT be used to inject natural-language instructions into FinBot's goals or system prompt; goal changes require a deployment by authorized engineering staff. / FinBot's decision logic is fixed by its deployment and MUST NOT be redefinable at runtime by any user, vendor, or caller."
src/routes/admin.py:90 — update_finbot_goals route has no authentication; its own comment reads 'MAJOR VULNERABILITY' and passes data['goals'] straight to finbot_agent.update_goals() src/services/finbot_agent.py:744 — update_goals() stores new_goals into config.custom_goals with no validation of content src/services/finbot_agent.py:70 — get_system_prompt() appends custom_goals under the header 'CUSTOM GOALS (OVERRIDE ABOVE IF CONFLICTING)'
Recommended Action
  • Require authentication and authorization on every /api/admin/* route; goal changes should not be a runtime HTTP operation — move them to a deploy-time artifact controlled by engineering staff.
  • Remove the custom_goals concatenation from get_system_prompt() so decision logic cannot be redefined from a config field.
