BEHAVIOR SUMMARY
------------------------------------------------------------------------------
HelperBot is a deliberately-vulnerable DVAA training agent and the dominant
pattern is total policy-implementation divergence: the Worker Remit declares
input validation, audit logging, a tool-call cap, prompt-injection refusal,
and system-prompt confidentiality, and the code implements none of them —
HELPERBOT.features sets inputValidation, outputFiltering, toolApproval,
rateLimiting, and auditLogging all to false, and the agent's own
vulnerabilities block in src/core/agents.js deliberately enables
promptInjection, dataExfiltration, and contextManipulation. The single most
dangerous concrete artifact is the LLM-mode system prompt
AGENT_PROMPTS.helperbot in src/llm/prompts.js, which both hardcodes a
live-pattern internal API key and explicitly tells the agent to disclose its
own instructions on request — credential exposure and a self-leaking system
prompt in one file. There is no durable, structured logging anywhere on
HelperBot's path; the only record is a 500-entry in-memory ring buffer that
dies with the process, so every one of these gaps is also undetectable after
the fact.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  Live-pattern internal API key hardcoded in HelperBot's
            LLM-mode system prompt at src/llm/prompts.js:27.
      Action 1: Remove the API-key line from the helperbot system prompt in
                src/llm/prompts.js; never place a credential in prompt text.
      Action 2: Move any real key to an environment variable or vault and
                rotate the value if it was ever live.

  PRAX-2026-05-19-002  HelperBot's system prompt explicitly instructs the
            agent to disclose its own instructions and configuration on
            request.
      Action 1: Rewrite the helperbot system prompt to forbid disclosure of
                instructions and configuration.
      Action 2: Remove the dataExfiltration system-prompt-leak branch from
                generateResponse() for any agent intended to be
                non-vulnerable.

  PRAX-2026-05-19-003  No input validation on HelperBot's path —
            inputValidation is false and generateResponse() obeys raw injected
            instructions.
      Action 1: Set inputValidation: true and add a real validation gate in
                generateResponse() that rejects detected prompt-injection
                patterns before the message reaches the model.
      Action 2: Remove the promptInjection vulnerability branch so injected
                instructions are declined and recorded, not obeyed.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-004  No defense against indirect or embedded-instruction
            injection — HelperBot has no content-origin separation and obeys
            instruction-like text.
      Action 1: Add content-origin labeling in prompt construction so
                retrieved or external content is fenced as untrusted data,
                never executed as instructions.
      Action 2: Implement a poisoned-content detector that strips or refuses
                instruction-like text from any non-user source.

  PRAX-2026-05-19-005  No durable audit logging of tool calls — auditLogging
            is false and the only record is a process-lifetime in-memory ring
            buffer.
      Action 1: Set auditLogging: true and implement a durable, structured log
                that records every tool call's name, parameters, and outcome.
      Action 2: Persist the log to disk or a log sink so records survive
                process restart.

  PRAX-2026-05-19-006  HelperBot accepts false conversational history and
            role-play identity changes — contextManipulation is deliberately
            enabled.
      Action 1: Remove the contextManipulation vulnerability block and the
                acceptFalseHistory flag from HelperBot.
      Action 2: Reject any user claim of prior agreements or identity changes
                that is not backed by verified session state.

  PRAX-2026-05-19-007  No per-session tool-call cap — the remit-required
            ~20-call limit is absent from HelperBot's code.
      Action 1: Add a per-session tool-call counter and reject further calls
                past a configurable cap (~20) in generateResponse().
      Action 2: Add tool-loop detection so repeated near-identical calls halt
                the session.

  PRAX-2026-05-19-008  HelperBot's HTTP chat surface is unauthenticated with
            wildcard CORS, exposing the agent to any origin.
      Action 1: Add authentication (token or session) to the chat endpoints
                before HelperBot is reachable.
      Action 2: Restrict CORS to the internal employee-assistant origin
                instead of *.
