BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is a sandbox whose isolation is the operator's
responsibility to configure, not a property the code enforces — and the
defaults lean unsafe. create_default_code_executor() silently downgrades from
Docker to the host-level LocalCommandLineCodeExecutor with only a
warnings.warn, and the local executor copies the full parent environment
(os.environ.copy()) into every child process, so a single execution sees the
caller's credentials — the remit's host-isolation and local-executor-approval
clauses have no code-level enforcement. The Docker path never sets
network_mode, CPU, or memory limits and merges caller-supplied extra_volumes
with no validation, so the remit's approval-required gates for mounts, egress,
and resource ceilings are absent. Compounding all of it, no executor writes
the per-execution audit log the remit mandates, and the
LocalCommandLineCodeExecutor docstring advertises regex sanitization against a
list of dangerous commands that does not exist anywhere in the implementation
— a control that is documented, relied upon, and absent.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  create_default_code_executor silently falls back from
            Docker to the host-level LocalCommandLineCodeExecutor, with only a
            runtime warning and no approval gate.
      Action 1: Make the Docker-to-Local fallback in
                create_default_code_executor() opt-in via an explicit
                allow_local_fallback=False default, raising instead of
                silently downgrading.
      Action 2: Require an explicit operator-supplied flag (or environment
                gate) before any code path constructs
                LocalCommandLineCodeExecutor in a non-interactive deployment.

  PRAX-2026-05-19-002  LocalCommandLineCodeExecutor runs LLM-generated code
            with the full parent process environment, exposing the caller's
            credentials to executed code.
      Action 1: Pass an explicitly constructed minimal environment (e.g. only
                PATH, HOME, LANG) to create_subprocess_exec instead of
                os.environ.copy().
      Action 2: Document that the local executor provides no credential
                isolation and must only run on hosts with no sensitive
                environment variables.

  PRAX-2026-05-19-003  No executor enforces or records the human-approval gate
            the remit mandates before the local host executor is used in
            production.
      Action 1: Add an explicit approved_for_host_execution: bool = False
                constructor parameter that raises unless the operator sets it.
      Action 2: Surface a structured, acknowledgeable approval event rather
                than a suppressible UserWarning.

  PRAX-2026-05-19-004  DockerCommandLineCodeExecutor merges caller-supplied
            extra_volumes into the container mount set with no validation or
            approval gate.
      Action 1: Validate every extra_volumes host path against an operator
                allow-list and reject mounts outside the configured work
                directory unless an explicit approval flag is set.
      Action 2: Treat mode: "rw" mounts and any mount of /var/run/docker.sock
                as requiring separate, logged approval.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-005  No executor writes the per-execution audit log the
            remit requires; only ad hoc operational logging exists.
      Action 1: Add a structured audit-log emission at the end of every
                execute_code_blocks call recording timestamp, executor kind,
                language, work dir, timeout, exit status, and a SHA-256 digest
                of the code (never the body).
      Action 2: Route the audit record to a durable, append-only sink rather
                than the Python logging root logger.

  PRAX-2026-05-19-006  Work-directory confinement is enforced only for the #
            filename: comment and does not defeat symlinks or absolute paths
            written inside the executed code.
      Action 1: Document explicitly that work-directory confinement applies
                only to the executor's own file placement, not to file
                operations performed by the executed code, and that the Local
                and Jupyter executors provide no filesystem sandbox.
      Action 2: For the container executors rely on the container boundary;
                for Local and Jupyter recommend OS-level sandboxing, since the
                executor cannot confine the executed code's file I/O.

  PRAX-2026-05-19-007  DockerCommandLineCodeExecutor starts containers with
            default networking and no egress allow-list, permitting
            unrestricted outbound traffic from executed code.
      Action 1: Default the Docker container to network_mode="none" and
                require an explicit operator-supplied network policy to enable
                egress.
      Action 2: Add a network allow-list parameter and treat any value beyond
                it as requiring logged human approval.

  PRAX-2026-05-19-008  No executor inspects LLM-generated code for
            sandbox-escape, privilege-escalation, or exfiltration
            instructions, and the local executor docstring claims sanitization
            that does not exist.
      Action 1: Remove or correct the LocalCommandLineCodeExecutor docstring
                claim of dangerous-command sanitization, which currently
                overstates the control to operators and auditors.
      Action 2: If escape/escalation detection is intended, implement it as an
                explicit pre-execution check; otherwise document that semantic
                code safety is entirely the caller's responsibility.

  PRAX-2026-05-19-009  JupyterCodeExecutor's timeout shields the running cell,
            so an over-time cell is abandoned rather than terminated.
      Action 1: On timeout, interrupt the Jupyter kernel (kernel interrupt
                request) or restart it so the over-time cell is actually
                terminated, not merely detached.
      Action 2: Drop the asyncio.shield so the execution task is cancelled
                when the wall-clock timeout fires.

  PRAX-2026-05-19-010  DockerCommandLineCodeExecutor sets no CPU, memory, or
            pids limits on the container, leaving execution
            resource-unbounded.
      Action 1: Add mem_limit, nano_cpus, and pids_limit parameters with
                conservative defaults and pass them to containers.create.
      Action 2: Gate any increase above the default ceiling behind an
                explicit, logged operator approval.
