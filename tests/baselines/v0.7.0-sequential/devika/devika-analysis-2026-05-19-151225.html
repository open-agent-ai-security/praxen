Executive Summary
Agent Remit (as declared)
Devika is an autonomous software-engineering agent that takes a high-level natural-language project request, decomposes it into a plan, researches via web search, writes code into a per-project workspace, runs it, debugs, and reports progress to the operator. It runs locally with a web UI and pluggable LLM providers. Per the remit Devika MAY plan, select actions, write code into the project workspace, perform web research, and call operator-configured LLM providers; it MUST NOT execute arbitrary shell commands outside a sandboxed runner, write outside the per-project work directory, install packages without confirmation, modify its own code, or exfiltrate credentials. Every action must be recorded to a per-project audit log and each run must enforce wall-clock and step-count caps.
Behavior Summary (as observed)

The dominant pattern is policy declared in documentation with zero enforcement in code, on the agent's single most dangerous capability. The remit and ARCHITECTURE.md both state code runs inside a sandboxed environment, yet Runner.run_code() passes LLM-generated command strings straight to subprocess.run on the host, and the two files named for the control — src/sandbox/firejail.py and src/sandbox/code_runner.py — are empty stubs.

This forms a complete, unbroken exploit chain: untrusted user prompts and untrusted web-search page text flow unsanitized into LLM context (the remit's injection-screening requirement has no implementing code), the model emits a commands list, and the runner executes it on the host with no approval gate, no per-command policy, no path confinement, and no step or wall-clock cap. Everything the remit forbids — sandbox escape, host filesystem access, package installs without confirmation — is reachable in one injection hop, and the agent keeps no per-project audit log to detect it after the fact.

Scope of Analysis
Python Flask + Flask-SocketIO application; devika.py exposes the HTTP and socket API on 0.0.0.0:1337. A single Agent class in src/agents/agent.py orchestrates twelve sub-agent prompt modules (planner, researcher, coder, runner, patcher, and others), each calling pluggable LLM clients via src/llm/llm.py. The Runner in src/agents/runner/runner.py executes LLM-produced shell command lists with subprocess.run directly on the host with cwd=project_path — there is no sandbox interposition, and src/sandbox/firejail.py and src/sandbox/code_runner.py exist but are 0-byte empty files. State and per-project knowledge share one SQLite DB. Provider credentials live in config.toml copied from sample.config.toml, which ships placeholder keys only. No test suite and no MCP configuration are present.

CRITICAL PRAX-2026-05-19-001 Runner executes LLM-generated shell commands directly via subprocess.run on the host — the declared sandboxed runner does not exist.
Implement Zero Trust LLM05 — Improper Output Handling ASI05 — Unexpected Code Execution (RCE)
Policy Rule — R-01, R-13 (Worker Remit):
"Devika MUST NOT execute arbitrary shell commands outside its sandboxed runner. / Devika MUST NOT execute LLM-produced code anywhere but inside the sandboxed runner."
src/agents/runner/runner.py:85 — subprocess.run(command_set, stdout=..., stderr=..., cwd=project_path) executes each command from the LLM-produced 'commands' list with no sandbox wrapper src/sandbox/firejail.py — File exists but is 0 bytes — the named firejail sandbox control is an empty stub src/sandbox/code_runner.py — File exists but is 0 bytes — planned code-runner isolation control was never implemented
Recommended Action
  • Implement src/sandbox/firejail.py (or a container-based runner) and route every subprocess.run call in runner.py through it before any execution path is reachable.
  • Until a real sandbox exists, gate Runner.run_code behind an explicit per-command human approval step.

HIGH PRAX-2026-05-19-006 File writes are not confined to the project work directory — coder and patcher join LLM-supplied file paths with no traversal check.
Implement Zero Trust LLM05 — Improper Output Handling ASI02 — Tool Misuse and Exploitation
Policy Rule — R-02, R-07 (Worker Remit):
"Devika MUST NOT write files outside the configured per-project work directory. / All file reads and writes MUST be confined to the current project's work directory, rejecting any path that escapes it."
src/agents/coder/coder.py:73 — save_code_to_project() does os.path.join(project_dir, project_name, file['file']) with file['file'] taken verbatim from LLM output — a '../' segment escapes the directory src/agents/patcher/patcher.py:74 — Patcher.save_code_to_project() repeats the same unchecked os.path.join pattern src/agents/coder/coder.py:55 — current_file is parsed from model text with no path normalization or containment check

HIGH PRAX-2026-05-19-007 Agent installs packages from LLM-generated commands with no user confirmation, and its own 31 dependencies are fully unpinned.
Manage Your Supply Chain LLM03 — Supply Chain ASI04 — Agentic Supply Chain Vulnerabilities
Policy Rule — R-03, R-15 (Worker Remit):
"Devika MUST NOT install system packages. / Devika MUST NOT install packages or pull in dependencies without explicit user confirmation, especially from unpinned versions or unverified registries."
src/agents/runner/prompt.jinja2:24 — The runner prompt's example output is literally 'pip3 install -r requirements.txt' — install commands are an expected runner output src/agents/runner/runner.py:85 — run_code() executes whatever install command the model returns with no confirmation step requirements.txt:1 — All 31 of Devika's own dependencies are unpinned — no version specifiers and no lock file
