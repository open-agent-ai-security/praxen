BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy declared in documentation with zero enforcement
in code, on the agent's single most dangerous capability. The remit and
ARCHITECTURE.md both state code runs inside a sandboxed environment, yet
Runner.run_code() passes LLM-generated command strings straight to
subprocess.run on the host, and the two files named for the control —
src/sandbox/firejail.py and src/sandbox/code_runner.py — are empty stubs. This
forms a complete, unbroken exploit chain: untrusted user prompts and untrusted
web-search page text flow unsanitized into LLM context (the remit's
injection-screening requirement has no implementing code), the model emits a
commands list, and the runner executes it on the host with no approval gate,
no per-command policy, no path confinement, and no step or wall-clock cap.
Everything the remit forbids — sandbox escape, host filesystem access, package
installs without confirmation — is reachable in one injection hop, and the
agent keeps no per-project audit log to detect it after the fact.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  Runner executes LLM-generated shell commands directly
            via subprocess.run on the host — the declared sandboxed runner
            does not exist.
      Action 1: Implement src/sandbox/firejail.py (or a container-based
                runner) and route every subprocess.run call in runner.py
                through it before any execution path is reachable.
      Action 2: Until a real sandbox exists, gate Runner.run_code behind an
                explicit per-command human approval step.

  PRAX-2026-05-19-002  Untrusted user prompts and web-retrieved page text
            enter LLM context with no injection screening, contradicting the
            remit's mandatory untrusted-input rule.
      Action 1: Add an instruction-injection screening pass on user prompts
                and on web-fetched content before either reaches any agent
                prompt template.
      Action 2: Label external content as untrusted in prompt construction and
                instruct sub-agents not to act on instructions found inside
                it.

  PRAX-2026-05-19-003  No human-in-the-loop approval gate before shell
            execution, package installs, or out-of-directory writes despite
            the remit requiring control to return to the user.
      Action 1: Insert an approval checkpoint in Runner.execute that emits the
                proposed command list to the operator and blocks on
                confirmation.
      Action 2: Maintain an allowlist of pre-approved commands and route
                anything outside it to human approval.

  PRAX-2026-05-19-011  Compound chain — unscreened external input flows to LLM
            context and then to unsandboxed host shell execution in a single
            injection hop.
      Action: Break the chain at every link: screen external input
              (PRAX-2026-05-19-002), sandbox execution (PRAX-2026-05-19-001),
              and require approval before run (PRAX-2026-05-19-003).

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-004  No wall-clock or step-count cap on an agent run — the
            remit's mandatory runaway-loop control is absent.
      Action: Add MAX_STEPS and MAX_WALLCLOCK config keys and enforce them in
              Agent.execute and subsequent_execute, halting and returning
              control to the user when exceeded.

  PRAX-2026-05-19-005  Operator API keys are stored in plaintext config.toml
            in the workspace and are readable by the running agent process.
      Action 1: Load provider credentials from environment variables or a
                secrets manager instead of a workspace file, and keep
                config.toml out of any path the agent can read.
      Action 2: Explicitly exclude config.toml and the repo root from ReadCode
                traversal.

  PRAX-2026-05-19-006  File writes are not confined to the project work
            directory — coder and patcher join LLM-supplied file paths with no
            traversal check.
      Action: After joining, resolve the path with os.path.realpath and reject
              any result whose common prefix is not the project directory —
              the pattern already used in project.py get_project_files.

  PRAX-2026-05-19-007  Agent installs packages from LLM-generated commands
            with no user confirmation, and its own 31 dependencies are fully
            unpinned.
      Action 1: Treat any install command emitted by the runner as requiring
                explicit human approval before execution.
      Action 2: Pin every dependency in requirements.txt and add a lock file.

  PRAX-2026-05-19-008  Project state and the knowledge base share one SQLite
            DB with no project scoping enforced — cross-project memory mixing
            is possible.
      Action: Add a project column to the Knowledge table and filter every
              get_knowledge and add_knowledge call by the active project.

  PRAX-2026-05-19-010  No per-project audit log of agent actions — logging is
            a single free-form file capturing only REST routes and errors.
      Action: Add a structured, append-only per-project audit log recording
              every plan step, file write, shell command, web fetch, and LLM
              call with timestamps.
