BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy-implementation divergence concentrated on the
CLI's two explicit safety promises — the unauthenticated-deploy confirmation
gate and the "only declared sources" guarantee. The anonymous-auth
confirmation prompt in _deploy is gated on frontend.enabled AND auth.provider
== "anonymous", yet bundler.py generates the permissive auth.py handler
whenever any [auth] provider is set — so a project with provider = "anonymous"
but no [frontend] section ships an open API with no warning and no
confirmation. Separately, _auto_wire_issues_board_if_hub provisions a
recurring cron-scheduled issues-agent on the platform after a hub-backed
deploy — an agent surface the developer never declared — and remote MCP
servers carried into the bundle are neither TLS-enforced nor
version/integrity-pinned, both of which the remit explicitly requires. The
supporting THREAT_MODEL.md is self-admittedly stale, so these gaps are also
undocumented.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  The unauthenticated-API confirmation gate fires only
            when [frontend].enabled is set, so an anonymous-auth project with
            no frontend ships an open API silently.
      Action: In commands.py, key the anonymous warning and confirmation
              prompt on config.auth is not None and config.auth.provider ==
              "anonymous" alone — drop the frontend.enabled precondition so
              every anonymous deploy is gated.

  PRAX-2026-05-19-002  deploy auto-provisions a recurring cron-scheduled
            issues-agent on the platform for hub-backed deploys — an agent
            surface the developer never declared.
      Action 1: Gate _auto_wire_issues_board_if_hub behind an explicit project
                declaration or an interactive operator opt-in.
      Action 2: Include the recurring issues-agent in the print_bundle_summary
                output so the operator sees the surface before the deploy
                proceeds.

  PRAX-2026-05-19-003  _validate_mcp_for_deploy rejects stdio MCP transports
            but never checks that http/sse server URLs use TLS, so a plaintext
            http:// MCP server passes validation into the bundle.
      Action: In _validate_mcp_for_deploy, add a validation error for any
              http/sse server whose url does not start with https://, so
              non-TLS MCP endpoints fail validation before the bundle ships.

  PRAX-2026-05-19-004  Remote MCP servers carried into the deployment bundle
            are copied verbatim with no version pin or integrity check, so the
            deployed agent loads whatever the URL serves at runtime.
      Action: Require an integrity reference — a pinned server version, an
              expected tool-manifest hash, or a signature — for each remote
              MCP server, and reject or warn at bundle time when one is
              absent.
