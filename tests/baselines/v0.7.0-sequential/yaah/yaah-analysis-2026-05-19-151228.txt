BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is policy-implementation divergence on the harness's
single central promise. The remit states that every generated agent
configuration MUST carry the same protective hook set; the code does not
deliver this. pkg/generator/hookmap.go has no Codex entry for PreToolUse or
PostToolUse, so CodexGenerator.GenerateHooks() emits a .codex/hooks.json
containing only SessionStart and Stop — the command guard, secret scanner,
linter, and comment checker are dropped for the Codex target, and
TestCodex_GenerateHooks_NoSupported codifies that drop as expected behaviour.
A second theme runs through the protections that do exist: they are advisory
or post-hoc rather than enforcing. The secret scanner and linter are
PostToolUse handlers that run after the write has already landed, the
yaah_check_command MCP tool only returns a safe/unsafe verdict with no
blocking, and there is no human-approval checkpoint for file writes or for
write/send/execute MCP tools — only catastrophic bash commands are
hard-blocked, and only via a small bypassable regex denylist.

CRITICAL FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-001  Codex-targeted configuration ships without the
            protective Pre/PostToolUse hooks — the command guard, secret
            scanner, linter, and comment checker are silently dropped.
      Action 1: Add Codex event names for PreToolUse and PostToolUse in
                pkg/generator/hookmap.go (Codex hooks.json supports them) so
                the protective handlers reach the Codex target.
      Action 2: If Codex genuinely cannot run Pre/PostToolUse hooks, make yaah
                generate fail loudly for the codex target rather than silently
                emitting a degraded config, and document the protection gap.

  PRAX-2026-05-19-002  No human-approval checkpoint exists for file writes or
            for write/send/execute MCP tools — only catastrophic bash commands
            are blocked.
      Action 1: Add a PreToolUse gate for Edit/Write/MultiEdit that requires
                explicit confirmation, or document that approval is delegated
                to the host agent's permission system and verify each
                generated config enables it.
      Action 2: Gate the write-capable MCP tool (yaah_planning_init) behind a
                confirmation or a dry-run/confirm two-step.

HIGH FINDINGS
------------------------------------------------------------------------------
  PRAX-2026-05-19-003  Hardcoded-credential detection is post-write only — the
            secret scanner runs on PostToolUse, after the edit has already
            been written to disk.
      Action 1: Treat PostToolUse blocking as detection-after-write and
                document that the file is already on disk; recommend the
                operator add a pre-commit secret scan as the actual
                before-accepted gate.
      Action 2: Add an entropy-based fallback pattern to catch credential
                formats not in the fixed list.

  PRAX-2026-05-19-004  No adversarial testing of the protections, no threat
            model, and no security-disclosure process.
      Action 1: Add an end-to-end test that generates config for every target
                and asserts the linter, command guard, secret scanner, and
                comment checker are all present and wired.
      Action 2: Publish a SECURITY.md with a disclosure process and a short
                threat model covering the harness's protective promises.

  PRAX-2026-05-19-006  No control prevents poisoning writes into
            agent-managed, session-loaded files (AGENTS.md, generated configs,
            .planning/STATE.md).
      Action 1: Add a handler or generator-side check that flags or blocks
                writes into agent-context files (AGENTS.md, generated
                settings, .planning/STATE.md) for human review.
      Action 2: Document which files are session-loaded and treat edits to
                them as requiring approval.

  PRAX-2026-05-19-010  Third-party MCP servers are launched via npx -y, which
            auto-installs the latest unpinned package on every run.
      Action 1: Pin third-party MCP packages to an exact version (e.g.
                @context7/mcp@<version>) and ideally an integrity hash, in the
                provider definitions under pkg/mcp/providers/.
      Action 2: Document a review/update process for bumping pinned MCP server
                versions.
