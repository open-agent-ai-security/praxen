BEHAVIOR SUMMARY
------------------------------------------------------------------------------
The dominant pattern is a control-plane that is correctly designed for the
SAAS deployment and structurally under-defended in the default OSS one. Every
V1 router carries the comment that protection is provided by
SetAuthCookieMiddleware, but that middleware lives only in
enterprise/server/middleware.py and is registered solely by
enterprise/saas_server.py; the OSS app_server/app.py registers no
authentication middleware at all, and utils/dependencies.get_dependencies()
returns an empty dependency list unless the SESSION_API_KEY env var is set.
The net effect: in a default local OSS run, the /api/v1/secrets,
/api/v1/sandboxes, /api/v1/conversations, and /api/v1/git endpoints answer
with no authentication — including the endpoints that read and write stored
git-provider tokens. The same fail-open shape recurs in
LocalhostCORSMiddleware, which returns True for every origin when no CORS
origins are configured. The remit's strongest behavioral guarantees — code
runs only in the sandbox, paths cannot escape the workspace, LLM tool-call
arguments are validated and clamped, per-session step and time caps — are real
but enforced in the external agent-SDK packages, so they cannot be verified at
this layer and are recorded as Enforcement-Not-Possible rather than as gaps.
